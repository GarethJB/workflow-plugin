# Implementation First Pipeline

구현 먼저 → 사후 테스트의 전통적 순서를 따르되, **커밋 단위의 커버리지 게이트**로 테스트 누락을 방지한다.

## 왜 커밋 단위 게이트가 필요한가

일반 구현은 가장 익숙하고 가벼운 방법론이지만, 명확한 안티패턴이 하나 있다: "일단 구현하고 나중에 테스트 쓰자"는 다짐이 **거의 항상 미이행**으로 귀결된다. 이유는 단순하다:

- 구현이 끝나는 순간 "동작 확인"이 되므로 성취감이 발생하고, 후속 테스트 작성의 동기가 급락한다.
- 다음 작업으로 넘어가면 이전 구현의 세부가 잊혀 테스트 작성이 어려워진다.
- 스프린트 압박·커밋 수 압박 속에서 "테스트는 다음 PR에"가 반복되며 미뤄진다.

방어는 **각 커밋마다 §13에 명시된 해당 계층의 테스트 커버리지를 확인**하는 게이트를 강제하는 것이다. 테스트를 반드시 같은 커밋에 포함해야 한다는 뜻은 아니지만, **커밋마다 "누락이 발생했는가"를 명시적으로 판단**해야 한다.

## 핵심 원칙

- **구현 후 테스트는 허용하되, 미루는 것은 허용하지 않는다.** 테스트는 구현과 같은 PR 또는 뒤따르는 단일 후속 커밋에 포함한다.
- **커밋마다 §13 대조.** 이번 커밋의 변경이 §13의 어느 계층에 해당하는지 식별하고, 해당 계층 테스트가 커버했는지 확인한다.
- **누락이 허용되는 유일한 경우**: §13에 "해당 없음"으로 명시된 계층에 대한 변경. 이때도 Changelog에 누락 사유를 남긴다.

## Phase 구조

일반 구현은 단일 Phase로 진행되지만, **커밋 게이트가 Phase 역할**을 수행한다.

| 구간 | 행동 | 체크 시점 |
|---|---|---|
| **구현** | 코드 작성 | 자유 |
| **커밋 전** | §13 대조 | 매 커밋 |
| **§14 AC 단위 완료 시** | 해당 AC를 검증하는 테스트 존재 확인 | AC 구현 완료 시점 |
| **전체 종료 시** | §13 계층별 커버리지 종합 점검 | Phase 종료 |

## 행동 규칙

### 구현 중

- 자유롭게 구현한다. 순서·스타일 제약 없음.
- 각 모듈·함수가 완성되면 **즉시 이번 변경이 §13의 어느 계층에 속하는지** 머릿속으로 태그한다 (단위 / 통합 / E2E / 해당 없음).

### 커밋 직전

아래 체크리스트를 수행한다:

1. **이번 커밋이 §13의 어느 계층과 관련되는가?**
   - 단위: 독립 함수·클래스 로직 변경
   - 통합: 모듈 경계·외부 API 통합 변경
   - E2E: 사용자 플로우 변경
   - 해당 없음: 문서·설정·포매팅 등
2. **해당 계층의 테스트가 이 커밋 또는 이전 커밋 중 하나에 포함되었는가?**
   - 예 → 커밋 진행
   - 아니오 → 세 가지 선택:
     - (a) 테스트를 먼저 작성하고 커밋에 포함한다.
     - (b) 의도적으로 미루는 경우 Changelog에 "테스트 미루기: [파일 또는 모듈] — 사유: [왜 지금 쓸 수 없는지] — 작성 예정: [시점 또는 조건]"을 기록하고 커밋한다. 이 기록은 추적 대상.
     - (c) §13 계획이 틀렸다고 판단되면 `spec-writing` 경로 C로 §13을 갱신한다.

**"나중에 반드시 쓸게"는 허용하지 않는다.** 최소한 "언제, 어떤 조건이 충족되면 쓴다"를 Changelog에 남겨야 미루기가 유효하다.

### §14 AC 단위 완료 시

어떤 AC-NN의 구현이 완료되었다고 판단되는 시점에:
- 해당 AC-NN을 검증하는 테스트가 존재하는지 확인.
- 없다면 위 커밋 직전 규칙과 동일한 선택.

### Phase 종료 (feature 5단계 구현 종료) 시

§13의 각 계층별로 계획된 테스트가 실제로 모두 작성되었는지 종합 점검한다:

- 단위: §13에 열거된 단위 테스트 항목이 실제로 존재하는가?
- 통합: 동일.
- E2E: 동일.

누락 항목이 남아 있으면 Phase 종료를 보류하고 작성한다. 이것이 feature 6단계(검증) 진입 전의 마지막 게이트다.

## 구조적 장치

### 잠금 파일

- `.implementation-first-active`

Phase 진입 시 생성 → Phase 종료 시 삭제. `.gitignore`에 추가.

### pre-commit 커버리지 경고 (권장)

Git pre-commit hook으로 커밋 diff에 **구현 파일만 있고 대응 계층 테스트 파일이 없으면 경고**하는 스크립트를 추가할 수 있다. 강제 차단보다는 알림 수준이 적절 — 개발자가 "이번엔 미룬다"를 명시적으로 선택할 수 있어야 한다.

hook 구현은 팀 결정 사항이며 이 skill의 범위 밖. 기본은 규약과 커밋 전 수동 체크로 운영.

### Changelog 마커

```
| 2026-04-20 | 일반 구현 Phase 진입 |
| 2026-04-20 | 커버리지 점검: AC-01~04 구현 완료, 단위 테스트 4건 작성 |
| 2026-04-20 | 테스트 미루기: SocialAuthAdapter(네이버) — 사유: SDK mocking 도구 확정 전 — 작성 예정: Flesh 통합 테스트 세션 |
| 2026-04-20 | 일반 구현 Phase 종료: §13 커버리지 종합 점검 완료 (단위 12/12, 통합 4/4, E2E 3/3) |
```

미루기 기록이 있으면 Phase 종료 전 **미루기 항목이 해소되었는지**를 반드시 확인한다. 해소되지 않았다면 해소 또는 Spec 개정을 먼저 진행한다.

## feature 파이프라인과의 연결

4단계에서 일반 구현이 선택되면 5단계(구현)는 이 skill의 규칙을 따른다. feature의 Spec↔코드 일치 게이트는 일반 규칙대로 작동한다:

- 코드가 Spec과 어긋나면 (1) 코드 수정 또는 (2) Spec 개정 중 선택.
- 커버리지 게이트와 일치 게이트는 독립적이다. 둘 다 매 커밋 직전에 체크.

## 완료 판정

- §13의 각 계층별로 계획된 테스트가 모두 존재한다.
- §14의 모든 AC에 대응하는 테스트가 존재한다 (§13의 어느 계층이든).
- Changelog의 모든 "테스트 미루기" 기록이 해소되었다.
- 잠금 해제 및 Phase 종료 커밋 완료.

이후 feature 6단계(검증)로 넘어간다.
