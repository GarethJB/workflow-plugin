# Feature Pipeline

Spec 중심 파이프라인으로 feature 개발을 구조적으로 조형한다.

## 핵심 원칙

- **Spec = SOT.** Spec이 소스코드의 단일 진실 공급원이다.
- **구조적 조형.** 이 skill은 워크플로우 자체의 형태를 강제한다. 단계를 안내하거나 추천하는 것이 아니라, 구조를 보증한다.
- **고정 구조, 가변 깊이.** 모든 작업이 동일한 7단계 파이프라인을 통과한다. 각 단계의 밀도만 작업 규모에 따라 달라진다.
- **Spec 수정 권한 분리.** Spec 문서의 내용 수정(섹션 1~14)은 `spec-writing` skill을 통해서만 수행한다. 이 skill(feature)은 라이프사이클 조작(status 전이, Changelog 기록)만 직접 수행한다. Spec 파일은 hook으로 보호되므로, 라이프사이클 조작 시에도 잠금 파일(`touch .spec-writing-active` → 수정 → `rm .spec-writing-active`)을 사용해야 한다.
- **구조화된 요청 입력.** 이 skill로 진입하는 사용자의 초기 요청이 `docs/structured-request.md`의 5필드(문제·맥락·기대 결과·제약·산출물 형식)를 충족하지 않으면, 1단계 진입 전에 누락 필드를 질문해 메운다. 프로젝트 전역 제약(CLAUDE.md)은 필드 2의 맥락으로 자동 참조된다.

## 파이프라인

```
[1. Spec 작성] → [2. Spec 승인] → [3. 계획] → [4. Practice 선택] → [5. 구현] → [6. 검증] → [7. 완료]
    draft           approved        approved      approved        implementing  implementing    shipped
```

**이 파이프라인은 무조건 순서대로 진행한다. 어떤 단계도 건너뛸 수 없다.**

## 진입

개발자가 feature를 시작하면, 현재 Spec의 상태를 확인한다.

### Spec 탐색 방법

1. `specs/` 디렉터리를 확인한다 (Glob 패턴: `specs/SPEC-*.md`).
2. 개발자가 특정 Spec을 지정했으면 해당 파일을 읽는다.
3. 여러 Spec이 존재하면 개발자에게 어떤 Spec으로 작업할지 물어본다.
4. Spec 파일의 YAML frontmatter에서 `status` 필드를 읽어 현재 상태를 파악한다.

### 상태별 진입 지점

- Spec이 없으면 → **1단계(Spec 작성)**부터 시작
- `draft` 상태의 Spec이 있으면 → **2단계(Spec 승인)**부터 시작
- `approved` 상태의 Spec이 있으면 → Spec의 Changelog를 확인한다:
  - 계획 확정 기록이 없으면 → **3단계(계획)**부터 시작
  - 계획 확정 기록은 있지만 Practice 선택 기록이 없으면 → **4단계(Practice 선택)**부터 시작
- `implementing` 상태의 Spec이 있으면 → **5단계(구현)**을 이어감

Changelog가 상태 복원의 핵심이다. Spec의 status가 `approved`인 상태에서 3단계와 4단계를 구분하려면 Changelog에 해당 단계의 완료 기록이 있는지를 확인해야 한다. 이것이 Changelog의 존재 이유 중 하나다.

## 1단계: Spec 작성

`spec-writing` skill을 호출하여 Spec을 작성한다.

**행위:**
- **복잡한 기능이면 brainstorming을 먼저 권장한다.** 아래 신호 중 2개 이상에 해당하면 `superpowers:brainstorming` skill을 먼저 실행하도록 개발자에게 제안한다:
  - 다수 모듈에 영향을 주거나 크로스 모듈 의존성이 있음
  - 외부 API / SDK / 서드파티 서비스 연동이 있음
  - 결제, 인증, 개인정보 등 비즈니스 민감도가 높음
  - 기존 코드/데이터 마이그레이션이 필요함
  
  > "이 기능은 규모가 있는 편입니다. `superpowers:brainstorming`으로 설계 방향을 먼저 확정하면 Spec 작성이 더 정확해집니다. brainstorming을 먼저 진행할까요?"
  
  brainstorming을 완료하면 생성된 설계 문서(`docs/superpowers/specs/*-design.md`)를 `spec-writing`의 입력으로 사용한다 (경로 A). brainstorming 설계 문서는 개발자가 주도한 설계 산출물이므로 §10.1(구조)의 "개발자 직접 기술" 규칙과 충돌하지 않는다 — 오히려 개발자 결정의 근거 자료가 된다.

- brainstorming을 생략하거나 완료 후: `skills/spec-writing/SKILL.md`를 Read하고, 그 지시를 따라 Spec 작성을 진행한다.
- Spec이 `draft` 상태로 생성되면 이 단계가 완료된다.

**산출물:** `specs/SPEC-XXX-<기능명>.md` (status: draft)

**다음 단계로의 전이 조건:** Spec 파일이 존재하고 status가 `draft`이다.

## 2단계: Spec 승인

작성된 Spec을 리뷰하고 승인한다.

**행위:**
- Spec 문서를 개발자에게 제시한다.
- 개발자에게 리뷰를 요청한다:
  > "Spec을 리뷰해주세요. 수정이 필요하면 말씀해주시고, 괜찮으면 승인해주세요."
- **미처리 Spec 불일치 플래그 조회:** 재승인(`feature` 경유로 기존 Spec을 수정한 뒤 재승인하는 경우)이면 해당 Spec의 `id`가 `related_spec`인 `debug-logs/DEBUG-*.md`의 frontmatter `spec_flags`를 스캔해 미처리 플래그를 확인한다. 미처리 플래그가 있으면 `spec-writing` 경로 C로 해소한 뒤 플래그를 제거하거나 해소 기록을 덧붙인다. 최초 승인이면 이 절차는 건너뛴다.
- 리뷰의 기본은 `spec-writing`의 "완성도 체크리스트"를 따른다. 아래는 2단계에서 추가로 확인할 파이프라인 관점의 포인트다:
  - 계약 그룹(§5 행위 명세, §6 에러 처리, §7 범위, §8 제약, §9 성능 기준)이 코드의 동작을 충분히 정의하는가?
  - 정의 그룹(§3 전제, §4 외부 의존성)의 전제가 현재 유효한가?
  - 수락 기준(§14)이 `AC-NN: [조건] 시 [관찰 가능한 결과]한다` 형식을 따르는가? MVP 스코핑이 필요한 경우 P1/P2 라벨이 부착되었는가?
  - 실현 그룹(§10 기술 설계, §11 플랫폼별 차이, §12 마이그레이션)에 "미정" 항목이 있는가? 있다면 `미정 — [이유] — 보완 조건: [trigger 또는 기한]` 형식으로 기록되어 있는지 확인한다. 형식 미준수 시 `spec-writing` 경로 C로 보완을 요청한다.
  - §10.1이 "미정"이면 3단계(계획) 진입 전에 개발자가 직접 정의해야 함을 안내한다 (spec-writing 규약상 10.1은 AI 제안 금지, 개발자 직접 기술 영역):
    > "기술 설계(10.1)가 아직 '미정'입니다. 이 영역은 개발자가 직접 정의해야 하는 부분으로, 3단계(계획) 진입 전에 채워야 합니다. 지금 정의하시겠습니까?"
- 수정 요청 시: Spec을 수정하고 다시 리뷰를 요청한다.
- 승인 시: 잠금 파일을 생성하고(`touch .spec-writing-active`), Spec의 frontmatter `status`를 `approved`로 변경하며 `updated` 날짜를 갱신하고 Changelog에 `status draft → approved (승인 완료)` 기록을 추가한 뒤 잠금 파일을 삭제하고(`rm .spec-writing-active`), 커밋한다.

**산출물:** spec.md (status: approved)

**다음 단계로의 전이 조건:** Spec의 status가 `approved`이다.

## 3단계: 계획

Spec을 기반으로 구현 계획을 수립한다.

**행위:**
- approved 상태의 Spec을 읽는다.
- **기술 설계(10.1)가 "미정"이면 계획 수립 전에 먼저 채운다.** `spec-writing` skill 경로 C를 통해 기술 설계 섹션을 보완한 후 계획 수립을 진행한다. 이때 10.1은 spec-writing 규약상 **개발자가 직접 기술하는 영역**이므로 skill은 개방형 질문("어떤 구조를 생각하고 계신가요?")만 던지고 AI가 단일 안을 단정하지 않는다. 10.2·10.3은 `[AI 제안]` 태그와 함께 복수 안 제안이 허용되므로 별개로 처리:
  > "Spec의 기술 설계(10.1)가 정의되지 않았습니다. 계획 수립 전에 개발자가 직접 정의해야 합니다. 10.2(인터페이스)·10.3(흐름)은 이후 AI가 복수 안으로 제안 가능합니다."
- Spec의 "실현" 그룹(기술 설계, 플랫폼별 차이, 마이그레이션)을 기반으로 구현 계획을 작성한다.
- 계획은 파일 단위, 모듈 단위의 작업 분해를 포함한다.
- `superpowers:writing-plans` skill이 사용 가능하면 이를 활용하여 계획을 작성한다. 사용 불가하면 대화 내에서 작업 목록으로 정리한다.
- 개발자에게 계획을 제시하고 확인을 받는다.
- **확인 후 잠금 파일을 생성하고(`touch .spec-writing-active`), Changelog에 "계획 확정" 기록을 남긴 뒤 잠금 파일을 삭제한다(`rm .spec-writing-active`).** 이 기록이 있어야 대화 단절 후 3단계를 건너뛸 수 있다.

**산출물:** 구현 계획 + Spec Changelog에 계획 확정 기록

**다음 단계로의 전이 조건:** 개발자가 계획을 확인하고, Changelog에 기록되었다.

## 4단계: Practice 선택 게이트

Spec의 깊이·§13 테스트 전략·§5 행위 명세를 기반으로 이번 구현에 적용할 개발 Practice를 선택한다. TDD는 여러 선택지 중 하나이며, 작업의 성격에 따라 더 적합한 Practice가 있다.

**행위:**
- Spec의 테스트 전략(§13)을 읽는다.
- **테스트 전략이 비어 있으면:** Practice 선택 전에 먼저 채운다. `spec-writing` skill 경로 C를 통해 테스트 전략 섹션을 보완한 후 게이트를 진행한다.
  > "Spec의 테스트 전략(§13)이 정의되지 않았습니다. Practice 선택 전에 먼저 정의하겠습니다."
- **테스트 전략이 있으면:** Spec의 깊이를 판단하고 Practice를 제안한다.

### Practice 후보

Practice는 **위험도 3단계**에 대응하도록 3종으로 운영한다. 혼용은 레이어별로 허용한다.

| Practice | 착수 방식 | 주된 입력 | 대응 위험도 · 적합한 상황 |
|---|---|---|---|
| **TDD** (Test-Driven Development) | 단위 테스트 먼저 → Red-Green-Refactor 사이클 | §13 단위 테스트 + §5 정상 흐름 + §14 AC | **고위험** — 비즈니스 로직 복잡, 회귀 민감, 금전·인증·데이터 정합성 |
| **Walking Skeleton** | 전체 플로우 얇게 관통 → 내부 점진 채움 | §5·§10.3 + §14 E2E | **중위험** — 크로스 모듈 통합 리스크, 미지의 연결 지점, 새 SDK·프로토콜 |
| **일반 구현** (Implementation First) | 구현 먼저 → 사후 테스트 | §13 전체 | **저위험** — 단순 기능, 내부 도구·프로토타입, 검증 가능 범위가 작음 |

### 깊이별 권장 매핑

`spec-writing` SKILL.md의 "깊이 판단 기준"과 연동한다:

| 깊이 | 위험도 | 기본 권장 |
|---|---|---|
| **가벼움** (단일 화면, 상태 없음, 외부 연동 없음) | 저 | 일반 구현 |
| **중간** (모듈 경계 있음, API 연동, 플랫폼 차이 경미) | 중 | Walking Skeleton |
| **무거움** (다수 모듈, 비즈니스 민감, 크로스 모듈 의존) | 고 | TDD |

기본값은 제안일 뿐이다. 개발자는 다른 Practice를 선택할 수 있고, **혼용**도 허용된다 (예: 관통 경로는 Walking Skeleton, 내부 도메인 로직은 TDD). 혼용 시에는 어느 레이어에 어떤 Practice를 적용했는지 모두 기록한다.

### 제안 형식

> "Spec 깊이 판단: [X] (위험도: [저|중|고]). §13 테스트 전략 개요: 단위 N / 통합 M / E2E K. **[Y] Practice를 권장합니다.** 다른 Practice를 선택하시겠습니까?
> (1) [권장] [Y]  (2) TDD  (3) Walking Skeleton  (4) 일반 구현  (5) 혼용"

### 기록

- 판단 결과를 Spec의 변경 이력(Changelog)에 기록한다:
  - 잠금 파일을 생성하고 (`touch .spec-writing-active`)
  - Changelog에 `Practice: [X] — 사유: [판단 근거]`를 기록하며 (혼용 시 레이어별로 명시)
  - 잠금 파일을 삭제한다 (`rm .spec-writing-active`)

**산출물:** Spec Changelog에 Practice 선택 기록

**다음 단계로의 전이 조건:** 개발자가 Practice를 선택하고, Changelog에 기록되었다.

### Practice별 5단계 구현 가이드

각 Practice는 전용 skill로 분리되어 있다. 4단계에서 선택된 Practice의 skill을 5단계에서 Read하여 해당 skill의 Phase 구조를 따른다. 아래는 요약이며, 구체 절차·잠금 파일·Changelog 형식은 각 skill의 SKILL.md를 따른다.

| Practice | skill 경로 |
|---|---|
| TDD | `skills/practice-tdd/SKILL.md` |
| Walking Skeleton | `skills/practice-walking-skeleton/SKILL.md` |
| 일반 구현 | `skills/practice-implementation-first/SKILL.md` |

#### TDD 선택 시
- `skills/practice-tdd/SKILL.md`를 Read하여 Red → Green → Refactor Phase 구조를 따른다.
- 테스트 작성과 구현은 **서로 다른 Phase**에서 수행한다. 같은 세션에서 병행하지 않는다.
- 단위 테스트가 Spec의 §5 행위 명세와 §14 수락 기준을 검증해야 한다.

#### Walking Skeleton 선택 시
- `skills/practice-walking-skeleton/SKILL.md`를 Read하여 Skeleton Phase와 Flesh Phase를 분리해 진행한다.
- Skeleton Phase에서는 비즈니스 로직·에러 처리·경계 조건을 작성하지 않는다. 관통 E2E는 항상 녹색을 유지한다.

#### 일반 구현 선택 시
- `skills/practice-implementation-first/SKILL.md`를 Read하여 커밋 단위 커버리지 게이트를 적용한다.
- 각 커밋 직전 §13 계층 대조를 수행하고, 미루기는 Changelog에 사유와 작성 예정 시점을 남겨야만 허용된다.

#### 혼용 선택 시
- 각 레이어·모듈별 선택 Practice를 Changelog에 명시한다 (예: "관통 경로: Walking Skeleton / 내부 도메인 로직: TDD / UI 컴포넌트: 일반 구현").
- 각 Practice의 구현 가이드를 해당 레이어에 적용한다.

## 5단계: 구현

Spec에 따라 코드를 작성한다.

**행위:**
- 잠금 파일을 생성하고(`touch .spec-writing-active`), Spec의 status를 `implementing`으로 변경한 뒤 잠금 파일을 삭제한다(`rm .spec-writing-active`).
- **구현 시작 전 Spec의 계약 그룹(§5 행위 명세, §6 에러 처리, §7 범위, §8 제약, §9 성능 기준)과 실현 그룹(§10 기술 설계, §11 플랫폼별 차이, §12 마이그레이션)을 전부 읽는다.** 이 단계를 건너뛰면 Spec ↔ 코드 일치 게이트가 반응적 체크로 전락한다.
- 4단계에서 선택된 Practice의 구현 가이드(4단계 "Practice별 5단계 구현 가이드")를 따른다. 각 Practice는 **착수 순서**만 다르고, 최종적으로 Spec의 계약·수락 기준을 모두 충족해야 한다.
- 구현 중 **Spec ↔ 코드 일치 게이트**를 적용한다 (아래 참조).

**산출물:** 소스 코드, 테스트 코드

**다음 단계로의 전이 조건:** 구현이 완료되었고, Spec ↔ 코드가 일치한다.

### Spec ↔ 코드 일치 게이트

구현 중 코드가 Spec과 다르게 흘러갈 경우, 다음 중 하나를 선택해야 한다:

1. **코드를 Spec에 맞게 수정한다.**
2. **Spec을 먼저 개정한 뒤 코드를 계속 작성한다.** Spec 내용 수정(섹션 1~14)은 반드시 `spec-writing` skill을 통해 수행한다 (`skills/spec-writing/SKILL.md`를 Read하고, 경로 C(기존 Spec 수정)를 따른다). 직접 편집하지 않는다.

**Spec과 코드가 불일치한 상태로 검증 단계에 진입할 수 없다.**

### 일치 게이트 체크 타이밍

모든 변경마다 체크하는 건 비효율적이다. 아래 시점에 Spec과 대조한다:

- **커밋 직전:** 커밋하려는 변경이 Spec의 행위 명세(5), 인터페이스(10.2)와 일치하는지 확인
- **새로운 파일/모듈 생성 시:** Spec의 기술 설계(10.1)에 정의된 구조와 맞는지 확인
- **예상과 다른 동작 발견 시:** 구현하다 "이건 Spec에 안 적혀 있는데?"라는 생각이 들면 즉시 확인

괴리를 감지하면 즉시 알린다:
> "현재 구현이 Spec의 [섹션명]과 다릅니다. (1) 코드를 Spec에 맞게 수정하거나, (2) Spec을 먼저 개정하시겠습니까?"

## 6단계: 검증

수락 기준(§14)이 실제로 통과하는지 확인한다.

### 왜 이 단계가 있는가

6단계는 **Spec의 SOT 권위가 실효화되는 단일 지점**이다. "코드가 Spec을 따른다"는 원칙이 선언으로 끝나지 않고 **관찰 가능한 사실로 전환**되려면, 이번 단계에서 AC가 하나씩 검증되어야 한다. 이 게이트가 없으면 Spec은 장식품이 되고, 이 게이트가 허술하면 SOT 주장은 신뢰를 잃는다. 따라서 6단계는 feature 파이프라인 전체에서 **가장 엄격한 규율이 요구되는 지점**이다.

### 입력 자산

- §14 수락 기준 (AC-01, AC-02, …) — 각 AC의 [P1]/[P2] 라벨 포함
- §13 테스트 전략 — AC ↔ 테스트 계층 매핑 (있으면 주석 참조)
- §9 성능 기준 — 수치 검증 대상
- 4단계에서 선택된 Practice — 검증 절차가 Practice별로 달라짐

### Practice별 특화 절차

4단계에서 선택된 Practice에 따라 6단계의 중점이 달라진다. 이미 Practice Phase에서 일부 검증이 끝난 경우 그 결과를 **재검증하지 않고 참조**한다.

| Practice | 6단계에서 중점 | Practice Phase 참조 |
|---|---|---|
| **TDD** | 통합·E2E 테스트 실행에 집중. 단위는 Green Phase 종료 시 이미 통과 확인됨. | Green/Refactor Phase 종료 커밋 |
| **Walking Skeleton** | 관통 E2E + 각 Flesh의 레이어 테스트 종합 실행. | 모든 Flesh 커밋 |
| **일반 구현** | §13 계층별 계획 테스트 전부 실행 + §14 AC별 검증. | 커밋 단위 커버리지 기록 |
| **혼용** | 레이어별 선택 Practice의 규칙을 각각 적용 후 결과 합산. | 각 레이어 Phase 기록 |

### 검증 절차

아래 순서로 진행한다.

#### 1. §14 × §13 매트릭스 작성

§14의 각 AC-NN이 §13의 어느 계층(단위/통합/E2E)에서 검증되는지 매핑한다. §13에 명시적 주석이 있으면 그대로 사용하고, 없으면 AC의 성격으로 판단:

| AC 성격 | 기본 계층 |
|---|---|
| 기능 동작 (예: "탭하면 이동한다") | 통합 또는 E2E |
| 내부 로직·계산 (예: "할인 금액이 차감된다") | 단위 |
| 성능 수치 (예: "5초 이내") | §9 측정 절차 |
| 코드 구조 (예: "모듈 분리가 기술 설계와 일치") | 코드 리뷰 |
| 플랫폼 호환 (예: "iOS 15+") | 플랫폼별 빌드/실행 |

#### 2. 테스트 실행 + 결과 수집

각 계층 테스트를 실제로 실행한다. 실행 결과를 AC-NN 단위로 모은다.

#### 3. 성능 기준 측정 (§9 대응 AC가 있을 때)

"측정했다"라는 말만으로는 부족하다. 아래 표준을 따른다:

| 항목 | 규정 |
|---|---|
| 측정 환경 | 기기·OS 버전·네트워크 상태·배터리·백그라운드 부하를 기록한다 (동일 조건 반복 가능하도록) |
| 반복 횟수 | 최소 10회 실행. 단일 측정은 통계적으로 무의미 |
| 분위수 | p50 / p95 / p99 중 §9에 명시된 것을 비교. 명시 없으면 p95 기본 |
| 비교 대상 | §9 테이블의 "기준" 값 |

#### 4. 회귀 범위 점검

- 이번 Spec의 AC 전부: 필수 실행
- `related_specs`로 연결된 Spec 중 **동일 모듈·동일 API를 공유하는 Spec의 AC**: 해당 영향권 AC 재실행
- 그 외: 스모크 수준 E2E 1건 (앱이 크래시 없이 기본 플로우를 통과하는지)

#### 5. 실패 분류와 라우팅

실패가 발견되면 원인을 분류해 **어느 단계로 돌아갈지**를 결정한다. 무조건 5단계 복귀가 아니다.

| 실패 원인 | 복귀 지점 | 조치 |
|---|---|---|
| 코드 결함 (Spec은 정확) | 5단계 (구현) | 코드 수정 후 6단계 재시도 |
| Spec 불일치 (코드가 옳을 수도) | `spec-writing` 경로 C → 2단계 재승인 | Spec을 개정하고 4단계부터 재진입. 단 깊이 가변 원칙에 따라 축약 진행 가능 |
| 환경·인프라 결함 | 환경 수정 후 6단계 재시도 | 복귀 아님. 환경 오염 여부는 Changelog에 기록 |
| Practice 선택 부적합 | 4단계 (Practice 선택) | 새 Practice 선택 후 해당 Phase 재진입 |

복귀 시 개발자에게 알린다:
> "AC-05 실패를 분석했습니다. 원인: [분류]. [복귀 지점]으로 돌아가서 [조치]를 진행할까요?"

#### 6. 부분 통과 판정 (P1/P2 기반)

전항 통과가 원칙이지만 다음 예외가 있다:

- **P1 실패 1건 이상**: shipped 차단. 실패 분류 후 복귀.
- **P1 전항 통과 + P2 실패 일부**: 개발자 판단으로 예외 허용 가능. 허용 시 Changelog에 사유·후속 작업 기한을 기록하고 7단계로 진행한다. 허용하지 않으면 P1 실패와 동일하게 복귀 처리.

### 보고 양식

각 회차의 검증 결과를 아래 형식으로 기록·보고한다.

```
## 검증 결과 (SPEC-XXX)
소요: 총 N분 / 재시도 M회
Practice: [선택된 Practice]

| AC | P | 계층 | 결과 | 세부 |
|---|---|---|---|---|
| AC-01 | P1 | E2E | ✅ | e2e/auto_login.spec.ts:15 · 1.2s |
| AC-05 | P1 | 통합 | ❌ | 기대 "메인 화면" / 실제 "로그인 화면" · AuthRepoTest.kt:40 · 원인: 코드 결함 → 5단계 복귀 |
| AC-12 | P2 | E2E | ✅ | e2e/duplicate_tap.spec.ts:8 |
| AC-15 | P2 | - | ⏸ | 스플래시 5초 측정: p95=5.4s (기준 5s 초과) · 원인: 코드 결함 → 5단계 복귀 |
| ...   |   |   |   |   |

총: 18/21 통과 (P1: 8/8 ✅, P2: 10/13)
회귀 점검: SPEC-003 AC-02 스모크 통과
결론: P1 실패 0건. P2 실패 3건에 대해 개발자 판단 필요.
```

### Changelog 기록

6단계 완료(또는 부분 통과 허용) 시 Spec §15에 기록한다:

```
| YYYY-MM-DD | 검증 완료: AC 18/21 통과 (P1 8/8, P2 10/13). 재시도 2회. 성능 p95: 스플래시 4.8s / SDK 27s. P2 실패 3건(AC-15, AC-19, AC-20)은 후속 작업 예정 (2026-05-15까지). |
```

실패로 복귀하는 경우에도 기록한다:

```
| YYYY-MM-DD | 검증 실패: AC-05 (코드 결함) → 5단계 복귀. AC-12 (Spec 불일치) → 2단계 재승인 예정. |
```

### 잠금 파일 절차

Changelog 기록도 Spec 파일 수정이므로 잠금 파일을 사용한다:

1. `touch .spec-writing-active`
2. Spec §15에 검증 기록 추가
3. `rm .spec-writing-active`

**산출물:** §14 대조 매트릭스, 성능 측정 결과, Changelog 검증 기록.

**다음 단계로의 전이 조건:** P1 전항 통과 + (P2 전항 통과 OR P2 실패에 대한 예외 승인 기록).

## 7단계: 완료

Spec을 최종 상태로 전이하고 feature를 마무리한다.

**전이 조건 확인:** 6단계의 전이 조건이 충족되었는가?
- P1 AC 전항 통과 ✓
- P2 AC 전항 통과 OR 미통과 항목에 대한 예외 승인 기록(후속 작업 기한 포함)이 Changelog에 남아 있음 ✓

위 조건이 충족되지 않았다면 7단계로 진입하지 않고 6단계로 되돌린다.

**행위:**
1. **패턴 환류 기록:** 이번 feature 구현 중에 **재사용 가능한 새 패턴·컨벤션**이 발견되었는지 개발자에게 확인한다. 후보는 예를 들어:
   - 새로운 에러 처리 컨벤션
   - 새로운 레이어 간 호출 규약 또는 모듈 경계
   - 새로운 외부 의존성 사용 패턴 (SDK 초기화·인증 토큰 처리 등)
   - 테스트 계층별 새 관행

   발견된 경우 현재 Spec의 Changelog에 `pattern-extracted: [패턴명 — 한 줄 요약]` 마커만 기록한다. **이 파이프라인은 CLAUDE.md를 직접 수정하지 않는다** — CLAUDE.md는 SOT 중 하나이므로, 반영 여부는 별도 검토·합의 과정을 통해 사람이 결정한다. 마커는 그 검토의 원천 자료가 된다.

   발견된 패턴이 없으면 "해당 없음"으로 기록하고 진행한다.

2. 잠금 파일을 생성하고(`touch .spec-writing-active`), Spec의 frontmatter `status`를 `shipped`로 변경하고 `updated` 날짜를 갱신하며 변경 이력(Changelog)에 `status implementing → shipped` 기록을 추가한 뒤 잠금 파일을 삭제한다(`rm .spec-writing-active`).
3. 커밋한다.
4. 개발자에게 완료를 알린다:
   > "SPEC-XXX `<기능명>`이 shipped 상태로 전이되었습니다. feature 파이프라인이 완료되었습니다."

**산출물:** spec.md (status: shipped), Changelog에 `pattern-extracted:` 마커 (있는 경우)

## Spec 상태 머신

```
draft → approved → implementing → shipped
                                     ↓
                                 superseded (by SPEC-XXX)
```

| 상태 | 의미 | 전이 조건 |
|---|---|---|
| `draft` | 작성 중 | spec-writing skill이 문서 생성 시 |
| `approved` | 리뷰 완료, 구현 가능 | 개발자 승인 |
| `implementing` | 구현 진행 중 | 5단계 진입 시 |
| `shipped` | 구현 완료, 코드와 일치 | 수락 기준 전항 통과 |
| `superseded` | 새 Spec으로 대체됨 | 후속 Spec이 approved될 때 |

### superseded 처리

기존 shipped Spec을 대체하는 새 Spec이 필요할 때:
1. 새 Spec을 `draft`로 작성한다.
2. 새 Spec의 `related_specs`에 기존 Spec ID를 포함한다.
3. 새 Spec이 `approved`될 때, 기존 Spec의 status를 `superseded`로 변경하고 `superseded_by`에 새 Spec ID를 기록한다.

## 역방향 전이

실무에서는 파이프라인을 거슬러 올라가야 할 때가 있다. 이것은 "스킵"이 아니라 "되돌아가기"이므로, 고정 구조 원칙에 위배되지 않는다.

### 허용되는 역방향 전이

| 현재 단계 | 되돌아갈 단계 | 조건 | 절차 |
|---|---|---|---|
| 5(구현) → | 3(계획) | 계획대로 구현이 불가능 | Changelog에 "계획 재수립 — 사유: [...]" 기록, 3단계 재진입 |
| 5(구현) → | 1(Spec 작성) | Spec이 근본적으로 틀림 | 현재 Spec을 `superseded`로 전이, 새 Spec을 `draft`로 작성 |
| 3(계획) → | 2(승인) | 계획 수립 중 Spec 결함 발견 | Spec을 수정하고 재승인 요청 |
| 6(검증) → | 5(구현) | 실패 원인이 **코드 결함** (Spec은 정확) | 코드 수정 후 6단계 재시도. 6단계 "실패 분류와 라우팅" 참조 |
| 6(검증) → | 2(승인) | 실패 원인이 **Spec 불일치** (코드가 옳을 수 있음) | `spec-writing` 경로 C로 Spec 개정 → 2단계 재승인 → 필요 시 4단계부터 재진입 (깊이 가변 원칙에 따라 축약 가능) |
| 6(검증) → | 4(Practice 선택) | **Practice 선택이 부적합**했다고 판명 | 새 Practice 선택 후 해당 Phase 재진입. Changelog에 이전 Practice와 변경 사유 기록 |
| 6(검증) 환경 오염 | (복귀 아님) | 실패 원인이 **환경·인프라** | 환경 수정 후 6단계 재시도. Changelog에 환경 오염 사실과 복구 기록 |

### 역방향 전이의 원칙

- **항상 Changelog에 기록한다.** 왜 되돌아가는지, 어떤 판단에 의한 것인지.
- **개발자에게 알린다.** 되돌아가기 전에 사유를 설명하고 확인을 받는다:
  > "구현 중 [문제]를 발견했습니다. [N단계]로 돌아가서 [수정 내용]을 반영하는 게 적절해 보입니다. 진행할까요?"
- **건너뛰기는 허용하지 않는다.** 5→1로 갈 때 2, 3, 4를 건너뛰는 게 아니라, 새 Spec으로 1→2→3→4→5를 다시 밟는다.

## 깊이 가변 원칙의 적용

이 파이프라인의 모든 단계는 작업 규모와 무관하게 항상 실행된다. 달라지는 것은 각 단계의 **밀도**다:

**가벼운 작업 (예: 단순 UI 수정):**
- Spec 작성: 각 섹션 한 줄. 전체 반 페이지.
- 계획: 간단한 작업 목록.
- Practice 선택: 일반 구현.
- 구현: 빠르게.
- 검증: 수락 기준 1-2개.

**무거운 작업 (예: 결제 로직):**
- Spec 작성: 행위 명세에 경계 조건 상세. 수 페이지.
- 계획: 모듈 단위 분해, 인터페이스 정의.
- Practice 선택: TDD (또는 관통 경로 Walking Skeleton + 내부 TDD 혼용).
- 구현: 선택된 Practice의 착수 순서를 따름.
- 검증: 수락 기준 다수, 성능 기준 포함.

**구조는 같고, 밀도만 다르다.**
