# Debug Pipeline

가설 기반 파이프라인으로 버그 수정을 구조적으로 조형한다.

## 핵심 원칙

- **구조적 조형.** 디버깅 워크플로우의 형태를 강제한다. 단계를 안내하거나 추천하는 것이 아니라, 구조를 보증한다.
- **고정 구조, 가변 깊이.** 모든 디버깅이 동일한 8단계 파이프라인을 통과한다. 각 단계의 밀도만 버그 규모에 따라 달라진다.
- **가설-검증 사이클.** 원인을 추측하고 바로 수정하는 것을 구조적으로 방지한다. 가설을 세우고 → 검증하고 → 확정된 원인만 수정한다. "아마 이것 때문일 거야" → 바로 수정 → 다른 문제 발생... 이 패턴은 시간을 낭비하고 코드를 오염시킨다. 가설-검증 사이클은 이 패턴을 구조적으로 막는다.
- **회귀 테스트 강제.** 회귀 테스트는 판단 게이트(사용자 선택)가 아닌 필수 단계다. 디버깅의 본질은 재발 방지이므로, 구조에 박혀 있다.
- **Spec 비침범.** debug skill은 Spec을 직접 개정하지 않는다. Spec과의 불일치를 발견하고 기록(플래그)만 한다. Spec은 `feature` 파이프라인의 정규 절차를 통해서만 개정된다. 이것은 Spec의 SOT 권위를 보호하기 위함이다. debug가 Spec을 직접 고치면, `feature`의 승인 절차를 우회하게 되고, Spec의 신뢰성이 훼손된다.

## 파이프라인

```
[1. 증상 기록] → [2. Spec 확인] → [3. 가설 수립] ⇄ [4. 검증] → [5. 원인 확정] → [6. 수정] → [7. 회귀 테스트] → [8. 완료]
```

3↔4단계 사이에 가설-검증 반복 루프가 존재한다. 가설이 기각되면 3단계로 돌아가 새 가설을 세운다. 이것은 허용된 루프이며, "스킵"이 아니라 "가설이 틀렸으므로 다시 세우는 것"이므로 고정 구조 원칙에 위배되지 않는다.

**이 파이프라인은 무조건 순서대로 진행한다. 어떤 단계도 건너뛸 수 없다.**

## 진입

개발자가 debug를 시작하면, 기존 디버깅 기록이 있는지 확인한다.

### 디버깅 기록 탐색

진행 중인 디버깅이 있는지 확인하려면:
1. 대화 컨텍스트에 이전 디버깅 단계의 기록이 있는지 확인한다.
2. `debug-logs/` 디렉터리에 진행 중인(`status: in_progress`) 기록이 있는지 확인한다 (Glob 패턴: `debug-logs/DEBUG-*.md`).
3. 해당 Spec의 Changelog에 진행 중인 버그 수정 기록이 있는지 확인한다.
4. 기록이 있으면 마지막 완료 단계 다음부터 재개한다.
5. 기록이 없으면 1단계부터 시작한다.

### Spec 유무에 따른 진입

- **Spec이 있는 기능의 버그:** 모든 단계가 Spec을 참조. 2단계에서 Spec 대조, 5단계에서 불일치 플래그.
- **Spec이 없는 기능의 버그 (레거시 등):** 2단계에서 "Spec 없음"으로 기록. 5단계에서 Spec 불일치 판단 건너뜀. 나머지 단계는 동일.

## 디버깅 기록 형식

모든 디버깅은 `debug-logs/DEBUG-XXX-<증상요약>.md`에 기록을 남긴다. 이 파일은 1단계에서 생성하고, 각 단계를 거치며 갱신한다.

```markdown
---
id: DEBUG-001
title: 증상 요약
status: in_progress | resolved
related_spec: SPEC-XXX (있으면)
spec_flags: [] (Spec 불일치 플래그 목록)
created: YYYY-MM-DD
resolved: YYYY-MM-DD (해결 시)
---

# 1. 증상
- 기대 동작:
- 실제 동작:
- 재현 조건:
- 영향 범위:

# 2. Spec 확인 결과
- Spec 유무:
- 불일치 여부:

# 3. 가설 이력
## 가설 1: [가설 내용]
- 검증 방법:
- 결과: 확인됨 / 기각됨 — [근거]

## 가설 2: [가설 내용] (기각 시)
- 검증 방법:
- 결과:

# 4. 확정 원인
- 코드 위치:
- 원인:
- 조건:

# 5. 수정 내용
- 변경 파일:
- 수정 요약:

# 6. 회귀 테스트
- 테스트 파일:
- 테스트 내용:

# 7. Spec 불일치 플래그 (해당 시)
- SPEC-XXX [섹션명]: [구체적 불일치 내용]
```

이 기록은 다음 목적을 가진다:
- **대화 단절 시 상태 복원:** `status: in_progress`인 기록의 마지막 채워진 섹션으로 재개 지점을 판단
- **디버깅 이력 추적:** 같은 영역에서 반복되는 버그 패턴을 발견할 수 있음
- **Spec 불일치 플래그 보존:** frontmatter의 `spec_flags`에 플래그가 기록되므로, 대화가 끊어져도 소실되지 않음

## 1단계: 증상 기록

버그의 증상을 구체적으로 기록한다.

**행위:**
- 디버깅 기록 파일(`debug-logs/DEBUG-XXX-<증상요약>.md`)을 생성한다. 기존 기록을 확인하여 다음 번호를 부여한다.
- 아래 항목을 기록한다:
  - **기대 동작:** 어떻게 동작해야 하는가
  - **실제 동작:** 실제로 어떻게 동작하는가
  - **재현 조건:** 환경, 입력, 순서, 빈도
  - **영향 범위:** 어떤 기능/화면에 영향을 주는가
- 개발자가 이미 정보를 제공했으면 추가 질문 없이 기록. 부족한 항목만 질문한다.

### 재현 불가 버그 처리

안정적 재현이 불가능한 버그(간헐적 크래시, Heisenbug 등)는 재현 조건 대신 아래를 기록한다:
- **발생 빈도:** 대략적 빈도 (예: "10회 중 1회", "특정 디바이스에서만")
- **발생 환경:** 발생 시의 환경 스냅샷 (OS 버전, 메모리 상태, 네트워크 상태 등)
- **수집 가능한 단서:** 크래시 로그, 스택 트레이스, ANR 리포트 등
- **재현 시도 기록:** 시도한 재현 방법과 결과

재현 조건 대신 이 항목들이 채워지면 전이 조건을 만족한다. 6단계(수정) 후 증상 해소 확인도 "재현 시도 N회 중 0회 발생"으로 대체한다.

**다음 단계로의 전이 조건:** 기대 동작, 실제 동작, 그리고 재현 조건 또는 재현 불가 단서가 기록되었다.

## 2단계: Spec 확인

해당 기능의 Spec이 존재하는지 확인하고, 존재하면 증상과 대조한다.

**행위:**
- `specs/` 디렉터리에서 관련 Spec을 탐색한다 (Glob 패턴: `specs/SPEC-*.md`).
- 개발자에게 어떤 기능의 버그인지 확인하여 관련 Spec을 찾는다.
- **Spec이 있으면:** 증상이 Spec의 행위 명세(5), 에러 처리(6)와 불일치하는지 대조한다.
  - "Spec 대로 동작하는데 기대와 다른 것" → Spec 자체의 결함 가능성
  - "Spec과 다르게 동작하는 것" → 코드의 결함
- **Spec이 없으면:** "Spec 없음"으로 기록하고 계속 진행한다.

**다음 단계로의 전이 조건:** Spec 확인 결과가 기록되었다.

## 3단계: 가설 수립

증상과 Spec 확인 결과를 바탕으로 코드를 탐색하고, 원인 가설을 세운다.

**행위:**

### 코드 탐색 (가설 수립 전)

좋은 가설은 코드를 본 뒤에 나온다. 가설을 세우기 전에 아래 탐색을 수행한다:

1. **증상 지점의 코드를 읽는다.** 에러 메시지, 크래시 스택 트레이스, 잘못된 출력이 발생하는 코드를 찾아 읽는다.
2. **최근 변경을 확인한다.** `git log`와 `git blame`으로 증상 지점 근처의 최근 변경을 확인한다. "이전엔 잘 되었는데 최근에 깨졌다"면 최근 커밋이 유력한 단서다.
3. **호출 체인을 추적한다.** 증상 지점에서 위로(caller) 또는 아래로(callee) 호출 체인을 따라가며 데이터 흐름을 파악한다.
4. **관련 테스트를 확인한다.** 기존 테스트가 있으면 읽어본다. 테스트가 이 시나리오를 커버하지 않을 수 있다.

모든 탐색을 할 필요는 없다. 증상의 성격에 따라 가장 유효한 탐색만 수행한다. 가벼운 버그면 1번만으로 충분할 수 있고, 복잡한 버그면 1-4번 모두 필요할 수 있다.

### 가설 세우기

탐색 결과를 바탕으로 가설을 세운다. 가설은 반드시 **구체적**이어야 하고, **검증 방법**이 함께 기록되어야 한다. 기존에 기각된 가설이 있으면 확인하여, 같은 가설을 반복하지 않는다.

**좋은 가설** — 위치, 조건, 동작이 구체적이고, 검증 방법이 명확:
- "`PaymentManager.applyCoupon()`에서 쿠폰 만료 여부를 체크하지 않아, 만료된 쿠폰이 할인에 적용된다. 검증: 만료된 쿠폰으로 `applyCoupon()`을 호출하면 할인이 적용되는지 확인."
- "`UserDefaults` 저장 시 키 이름이 `isDarkMode`와 `darkMode`로 불일치한다. 검증: 두 키를 모두 읽어서 값이 다른지 확인."

**나쁜 가설** — 추상적이거나 검증 방법 없음:
- "아마 이 함수가 문제일 것" (어떤 조건에서 어떻게?)
- "네트워크 관련 문제인 것 같다" (어떤 요청? 어떤 응답? 어떤 조건?)

### 복합 원인 가능성

하나의 버그에 여러 원인이 결합된 경우가 있다 (예: 조건 A와 조건 B가 동시에 만족될 때만 발생). 가설을 세울 때 "이것만이 원인인가, 다른 조건이 결합되어야 발생하는 건 아닌가?"를 함께 고려한다. 6단계(수정) 후 증상이 부분적으로만 개선되면, 복합 원인을 의심하고 3단계로 돌아가 추가 원인에 대한 새 가설을 세운다.

**다음 단계로의 전이 조건:** 구체적 가설과 검증 방법이 기록되었다.

## 4단계: 검증

가설의 검증 방법을 실행한다.

**행위:**
- 가설에 기록된 검증 방법을 실행한다: 로그 확인, 브레이크포인트, 단위 테스트 작성/실행, 값 추적 등.
- 검증 결과를 기록한다:
  - **"가설 확인됨 — 근거: [확인 결과]"** → 5단계로 진행
  - **"가설 기각됨 — 이유: [기각 근거]"** → 3단계로 돌아가 새 가설을 세운다

### 가설-검증 반복

```
[3. 가설 수립] → [4. 검증] → 확인됨 → [5. 원인 확정]
                     ↓
                  기각됨 → [3. 가설 수립] (새 가설)
```

기각된 가설은 디버깅 기록의 "가설 이력"에 남겨 같은 가설을 반복하지 않는다. 기각 기록은 새 가설을 세울 때 참고 자료가 된다 — "이미 네트워크 타임아웃은 아님을 확인했으므로, 파싱 로직을 살펴본다."

**다음 단계로의 전이 조건:** 가설이 확인되었다.

## 5단계: 원인 확정

검증된 가설을 원인으로 확정한다.

**행위:**
- 원인을 디버깅 기록에 기록한다:
  - **어떤 코드가** (파일, 함수, 줄 번호)
  - **왜** (로직 오류, 누락, 잘못된 조건, 타이밍 문제 등)
  - **어떤 조건에서** (입력, 상태, 타이밍)
  - 버그를 발생시키는가

### Spec 불일치 판단

Spec이 있으면 원인이 Spec 불일치에 해당하는지 판단한다:

- **Spec과 코드가 불일치** (코드가 Spec과 다르게 동작): 플래그를 기록한다.
  > "Spec 불일치 발견: SPEC-XXX [섹션명] — [구체적 불일치 내용]. feature 파이프라인에서 Spec 개정 필요."

- **Spec 자체에 결함** (Spec대로 동작하지만 그것이 잘못됨): 플래그를 기록한다.
  > "Spec 결함 발견: SPEC-XXX [섹션명] — [구체적 결함 내용]. feature 파이프라인에서 Spec 개정 필요."

- Spec이 없으면 이 판단을 건너뛴다.

**플래그 저장:** 플래그는 두 곳에 기록한다:
1. 디버깅 기록 파일의 "7. Spec 불일치 플래그" 섹션
2. 디버깅 기록 파일의 frontmatter `spec_flags` 필드

frontmatter에 기록하는 이유: `debug-logs/DEBUG-*.md`의 frontmatter를 스캔하면, 미처리된 Spec 불일치 플래그를 일괄 조회할 수 있다. `feature` 파이프라인에서 Spec을 개정할 때 이 목록을 참조한다.

**다음 단계로의 전이 조건:** 원인이 확정되고 기록되었다.

## 6단계: 수정

확정된 원인에 대한 수정 코드를 작성한다.

**행위:**
- 수정 범위는 확정된 원인에 한정한다. 주변 코드 정리나 리팩토링은 하지 않는다. 수정과 개선을 섞으면, 수정이 문제를 해결했는지 판단하기 어려워진다.
- 수정이 1단계의 증상을 해결하는지 재현 조건으로 확인한다.
- 재현 불가 버그의 경우: "재현 시도 N회 중 0회 발생"으로 확인한다.
- 수정 후에도 증상이 해소되지 않으면 원인이 잘못 확정되었거나 복합 원인일 가능성이 있다. 3단계(가설 수립)로 돌아간다.
- 수정 내용을 디버깅 기록에 기록한다.

**다음 단계로의 전이 조건:** 수정 코드가 작성되었고, 증상이 해소되었다.

## 7단계: 회귀 테스트 (강제)

이 버그가 재발하지 않음을 검증하는 테스트를 작성한다.

**이 단계는 스킵할 수 없다.** 오타 한 글자 수정이라도 회귀 테스트를 작성한다. 깊이가 가변될 뿐이다 — 오타면 문자열 검증 테스트 하나, 레이스 컨디션이면 동시성 테스트 세트.

**행위:**
- 1단계의 재현 조건을 반영한 테스트를 작성한다.
- 테스트가 수정 전 코드에서 실패하고, 수정 후 코드에서 통과하는지 확인한다 (가능한 경우).
- 경계 조건이 원인이었다면 관련 경계 조건 테스트도 추가한다.
- 회귀 테스트 작성 중 수정의 부작용을 발견하면 6단계(수정)로 돌아간다.
- 테스트 파일과 내용을 디버깅 기록에 기록한다.

**다음 단계로의 전이 조건:** 회귀 테스트가 작성되고 통과했다.

## 8단계: 완료

디버깅 기록을 정리하고 마무리한다.

**행위:**
- 디버깅 기록 파일의 `status`를 `resolved`로, `resolved` 날짜를 기입한다.
- 해당 Spec이 있으면 Spec의 Changelog에 버그 수정 기록을 남긴다:
  - "[날짜] 버그 수정: [증상 요약] — 원인: [원인 요약] (DEBUG-XXX 참조)"
- 커밋한다 (수정 코드 + 회귀 테스트 + 디버깅 기록).
- 개발자에게 완료를 알린다.

Spec 불일치 플래그가 있는 경우:
> "버그 수정이 완료되었습니다 (DEBUG-XXX). SPEC-XXX의 [섹션]에 불일치가 발견되었으므로, feature 파이프라인에서 Spec 개정이 필요합니다."

Spec 불일치 플래그가 없는 경우:
> "버그 수정이 완료되었습니다 (DEBUG-XXX)."

## 역방향 전이

### 허용되는 역방향 전이

| 현재 단계 | 되돌아갈 단계 | 조건 | 절차 |
|---|---|---|---|
| 4(검증) → | 3(가설 수립) | 가설 기각됨 | 기각 기록 남기고 새 가설 수립 |
| 6(수정) → | 3(가설 수립) | 수정 후에도 증상 미해소 (오진 또는 복합 원인) | "원인 재검토 — 사유: [...]" 기록, 3→4→5 재진행 |
| 7(회귀 테스트) → | 6(수정) | 회귀 테스트 중 수정 부작용 발견 | 부작용 기록, 수정 보완 |

### 역방향 전이의 원칙

- 항상 기록한다. 왜 되돌아가는지, 어떤 판단에 의한 것인지.
- 기각된 가설은 기록으로 남겨 같은 가설을 반복하지 않는다.
- 건너뛰기는 허용하지 않는다. 6→3으로 갈 때 4, 5를 다시 밟는다.

## 깊이 가변 원칙의 적용

이 파이프라인의 모든 단계는 버그 규모와 무관하게 항상 실행된다. 달라지는 것은 각 단계의 **밀도**다.

**가벼운 버그 (예: 오타, 단순 로직 오류):**
- 증상 기록: "버튼 텍스트가 '확인'이어야 하는데 '확이'으로 표시됨"
- Spec 확인: "해당 Spec 없음"
- 가설: 코드 탐색(1번만) → "`MainView.swift:42`의 문자열 오타" → 검증: 확인 → 1회 사이클
- 수정: 한 줄
- 회귀 테스트: 문자열 검증 테스트 1개
- 전체 과정: 수 분

**무거운 버그 (예: 간헐적 크래시, 레이스 컨디션):**
- 증상 기록: 재현 불가 → 발생 빈도, 환경 스냅샷, 크래시 로그 기록
- Spec 확인: 행위 명세와 대조, 경계 조건 확인
- 가설: 코드 탐색(1-4번 전부) → 여러 가설 → 기각 → 새 가설... 3-4회 반복
- 수정: 다수 파일
- 회귀 테스트: 통합 테스트 + 경계 조건 테스트 + 동시성 테스트
- 전체 과정: 수 시간~수 일

**구조는 같고, 밀도만 다르다.**
