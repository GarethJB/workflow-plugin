# Walking Skeleton Pipeline

전체 플로우를 얇게 관통하는 스텁부터 시작해 각 스텁을 점진적으로 실제 구현으로 교체한다. **통합 리스크를 조기에 드러내고 방어**한다.

## 왜 Phase 분리가 필요한가

Walking Skeleton의 가치는 "가장 먼저 확인되는 것이 통합 가능성"이라는 데 있다. 미지의 연결 지점(외부 SDK, 서버 계약, OS API, 빌드 설정)이 많을 때, **스켈레톤 Phase에서 실제 로직을 함께 구현**하면:

- 로직 구현에 시간이 소모되어 통합 확인이 뒤로 밀림
- 로직 버그와 통합 버그가 섞여 원인 판별이 어려워짐
- 결국 "전통적 빅뱅 통합"과 차이가 없어짐

방어는 **스켈레톤 Phase에서 로직 작성을 구조적으로 금지**하고 **관통 테스트가 통과하는 상태를 유지하는 규칙을 Phase마다 강제**하는 것이다.

## 핵심 원칙

- **스켈레톤 = 관통만 보증.** 각 단계가 "호출되고 응답을 반환"하는 것만 확인한다. 로직은 없다.
- **관통 테스트는 녹색을 유지한다.** 스텁 교체 중에도 관통 E2E는 항상 통과해야 한다. 실패 상태로 커밋 금지.
- **Phase 전환 = 커밋 경계.** 스켈레톤 완성 커밋 → Flesh 1 → Flesh 2 → … → 종료.
- **Changelog로 교체 진행을 기록.** 각 스텁이 실제 구현으로 교체된 시점을 남긴다.

## Phase 구조

| Phase | 입력 | 출력 | 수정 허용 | 수정 금지 | 전환 조건 |
|---|---|---|---|---|---|
| **Skeleton** | §5 전체 플로우 + §14 E2E 최소 1개 | 각 단계가 스텁으로 호출되는 얇은 관통 구현 + 관통 E2E | 모든 구현 파일 (스텁 수준만) | 비즈니스 로직, 에러 처리, 경계 조건 | 관통 E2E 통과 + 스텁만 있는지 체크 + 커밋 |
| **Flesh N (각 스텁 교체)** | 스켈레톤 + 교체 대상 Spec 요구사항 | 해당 스텁을 실제 구현으로 교체 | 해당 모듈 | 관통 E2E 실패 상태로 커밋 | 관통 E2E 여전히 통과 + 커밋 |

Flesh Phase는 스텁 수에 따라 여러 번 반복된다 (Flesh 1, Flesh 2, …). 한 Flesh에서 한 스텁씩 교체하는 것이 기본이나, 연관 스텁은 묶을 수 있다.

## Phase별 행동 규칙

### Skeleton Phase

- §5 행위 명세의 각 단계(예: "스플래시 진입", "자동 로그인 판별", "소셜 인증", "isRegistered 호출", "login 호출") 각각에 대해 최소한의 함수·클래스를 둔다.
- 각 구현의 본체는 **다음 중 하나**여야 한다:
  - 하드코딩된 성공 값 반환
  - `TODO: Flesh Phase에서 구현` 주석 + 최소 리턴
  - 외부 의존성(§4) 호출이 필요하면 모의 응답 반환
- §14에서 최소 1개의 E2E AC(전체 플로우 관통)를 선택해 테스트로 작성한다. 이 테스트가 스텁 상태로도 통과해야 한다.
- **금지**:
  - 경계 조건 처리 (§5의 경계 조건·§6 에러 처리)
  - 비즈니스 규칙 적용 (분기·계산·검증)
  - 실제 외부 호출 (스텁으로 대체)
- 스켈레톤이 너무 얇아 "의미 없어 보이는" 것이 맞다. 이것이 원칙이다.

### Flesh Phase

- 이번 Phase에서 교체할 스텁 1~몇 개를 선택한다. Changelog에 명시:
  > "Flesh 2 진입: SocialAuthAdapter(네이버) 실제 구현으로 교체"
- 교체는 다음 순서로:
  1. 스텁이 있던 자리에 실제 구현 작성
  2. 단위 테스트·통합 테스트를 필요에 따라 추가 (혼용 시 해당 skill 규칙 적용)
  3. 관통 E2E 재실행 → 여전히 통과 확인
  4. 통과 상태로 커밋
- 관통 E2E가 실패하면 **커밋하지 않는다**. 복구가 Phase의 책임이다.
- 한 Flesh Phase에서 여러 스텁을 교체할 때도 위 순서를 따른다. 교체 후 실패하면 직전 커밋으로 되돌리고 더 작은 단위로 재시도.

## 구조적 장치

### 잠금 파일

- `.walking-skeleton-phase-active` (값: `skeleton` 또는 `flesh-N`)

파일은 하나만 존재한다. Phase 전환 시 내용을 바꾸거나 새로 쓴다. `.gitignore`에 추가.

### 고아 잠금 확인

시작 시 잠금 파일 내용을 확인해 어느 Phase인지 파악하고 재개 또는 정리 여부를 개발자에게 확인한다.

### 스켈레톤 로직 부재 체크리스트 (Skeleton Phase 종료 전)

스켈레톤 커밋 전 각 스텁을 훑어 아래를 확인한다:

- 본체가 한 줄 return (하드코딩) 또는 TODO + 기본값만 포함한다.
- 조건문(`if`, `when`, `switch`)이 없거나 있어도 그 내용이 "TODO" 표기뿐이다.
- 외부 호출이 모의 응답으로 대체되었다.

하나라도 실제 로직이 포함되어 있으면 Flesh Phase로 이관한다.

### 관통 테스트 유지 게이트 (Flesh Phase 매 커밋 전)

커밋 전 관통 E2E를 실제 실행해 통과 확인. 실패 시 커밋 금지. 이 규칙은 Git pre-commit hook으로 강제하는 것을 권장한다 (팀 결정).

### Changelog 마커

```
| 2026-04-20 | Walking Skeleton Skeleton Phase 완료: 관통 E2E 1건 통과, 스텁 N개 |
| 2026-04-20 | Walking Skeleton Flesh 1/N 완료: SocialAuthAdapter(네이버) 교체 |
| 2026-04-20 | Walking Skeleton Flesh 2/N 완료: AuthRepository.checkSession 교체 |
| ... |
| 2026-04-20 | Walking Skeleton 종료: 모든 스텁 교체 완료 (grep TODO 0건) |
```

## feature 파이프라인과의 연결

4단계에서 Walking Skeleton이 선택되면 5단계(구현)는 이 skill의 Phase 구조를 따른다. feature의 Spec↔코드 일치 게이트:

- **Skeleton Phase에서의 불일치는 의도된 것**이다 (로직이 없으므로 완전한 계약 이행은 불가능). 이 Phase에는 게이트를 느슨하게 적용하고, Flesh Phase 완료 후 재점검한다.
- **Flesh Phase에서의 불일치**는 일반 규칙대로 처리 (코드 수정 또는 Spec 개정).

혼용 허용: 각 Flesh Phase 내부에서 TDD·ATDD·일반 구현을 자유롭게 선택할 수 있다. 선택 방법론을 해당 Flesh의 Changelog 항목에 명시한다.

## 완료 판정

- 프로젝트 내 모든 스텁이 실제 구현으로 교체됨 (`grep -r "TODO: Flesh" src/` 0건 또는 팀 합의 기준).
- 관통 E2E + §13 계획 테스트가 모두 통과.
- Changelog에 Walking Skeleton 종료 기록.

이후 feature 6단계(검증)로 넘어간다.
