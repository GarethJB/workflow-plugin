# TDD Pipeline

Red → Green → Refactor 사이클을 **Phase 분리**로 강제해 TDD의 가치를 보호한다.

## 왜 Phase 분리가 필요한가

TDD의 핵심 가치는 "테스트가 Spec의 계약을 기계적으로 번역한 것"이라는 데 있다. 그러나 **같은 세션에서 테스트와 구현을 번갈아 작성**하면 테스트 작성자가 구현을 의식하게 되어, 의도치 않게 다음 증상이 발생한다:

- **구현하기 쉬운 테스트**만 작성됨 (테스트가 내부 구조에 편향)
- 테스트가 Spec보다 **구현 의도**를 기록하게 됨
- 실패-통과 경계가 흐려져 "테스트가 실제로 무엇을 검증하는지" 불명확해짐

이 증상은 자발적 주의력만으로 막기 어렵다. **Phase를 분리해 컨텍스트를 물리적으로 갈라놓는 것**이 실효적 방어다.

## 핵심 원칙

- **테스트 = Spec의 번역.** 테스트는 §14 수락 기준과 §5 행위 명세를 검증 가능한 형태로 옮긴 것이지, 구현 편의의 반영이 아니다.
- **Phase 간 권한 격리.** 각 Phase에서 수정 가능한 파일 범주가 다르다. 침범은 잠금 파일 + hook으로 차단한다.
- **Phase 전환 = 커밋 경계.** 현재 Phase의 산출물을 커밋해야 다음 Phase로 진입할 수 있다. 커밋이 실질적 세션 리셋 역할을 한다.
- **Changelog 마커.** Spec §15에 각 Phase 전환을 기록한다. 대화 단절 후 상태 복원의 단서가 된다.

## Phase 구조

| Phase | 입력 | 출력 | 수정 허용 | 수정 금지 | 전환 조건 |
|---|---|---|---|---|---|
| **Red** | Spec §5, §13 단위, §14 AC | 실패하는 테스트 코드 | 테스트 파일 | 구현 파일 (읽기 최소화) | 모든 테스트가 실제로 실패하는지 확인 후 커밋 |
| **Green** | 테스트 + Spec | 테스트를 통과시키는 **최소** 구현 | 구현 파일 | 테스트 파일 | 모든 테스트 통과 후 커밋 |
| **Refactor** | Green 상태 코드 | 구조 개선된 코드 | 구현 파일 | 테스트 파일, 새 기능 추가 | 모든 테스트 여전히 통과 후 커밋 |

### Phase 전환 흐름

```
[Red 진입] → 잠금 파일 .tdd-phase-red-active 생성
           → 테스트 작성
           → 테스트 실행 → 모두 실패 확인
           → Red 상태 커밋
           → 잠금 해제
           → Changelog "TDD Red 완료: N개 테스트 실패 확인" 기록
           → Green 진입

[Green 진입] → 잠금 파일 .tdd-phase-green-active 생성
             → 최소 구현
             → 테스트 실행 → 모두 통과 확인
             → Green 상태 커밋
             → 잠금 해제
             → Changelog "TDD Green 완료: N개 테스트 통과" 기록
             → (필요 시) Refactor 진입

[Refactor 진입] → 잠금 파일 .tdd-phase-refactor-active 생성
                → 구조 개선
                → 테스트 실행 → 여전히 모두 통과 확인
                → Refactor 커밋
                → 잠금 해제
                → Changelog "TDD Refactor 완료" 기록
                → 다음 AC 묶음으로 Red 진입 (또는 종료)
```

## Phase별 행동 규칙

### Red Phase

- Spec의 어느 AC·행위 명세 항목을 검증할지 먼저 명시한다 (예: "AC-05, AC-06, AC-07").
- 테스트 이름·주석에 해당 AC-NN을 포함한다 (예: `test_ac_05_login_redirects_to_main`).
- 구현 코드에서 사용할 함수·클래스의 **시그니처만** 구상하고 테스트에 반영한다. 본체 구현은 쓰지 않는다.
- 테스트를 실행해 **실제로 실패**하는지 확인한다. 컴파일 실패도 실패로 간주하지만, 가능하면 컴파일 통과 + 단언 실패의 형태가 이상적이다.
- 여러 AC를 한 번에 Red로 작성할 수 있다. 반드시 AC 하나씩일 필요는 없다. 다만 테스트 건수가 많으면 Green Phase가 커지므로 실용적 묶음으로 나누는 것을 권장한다.

### Green Phase

- **테스트 파일을 수정하지 않는다.** 테스트가 잘못되었다고 느끼면 Green을 중단하고 Red로 복귀한다.
- "최소 구현"의 의미: 테스트를 통과시키는 데 필요한 코드만 작성한다. Spec에 있더라도 현재 Red에 없는 케이스는 구현하지 않는다 (다음 Red 묶음에서 다룬다).
- 모든 테스트가 통과하면 Phase 종료. 리팩터링 충동은 Refactor Phase로 미룬다.

### Refactor Phase

- 선택적이다. 중복·네이밍·구조에 개선 여지가 있을 때만 진입한다.
- 규칙: **테스트를 수정하지 않고, 새 동작을 추가하지 않고, 모든 테스트가 계속 통과**해야 한다.
- 마이크로 커밋 권장 — 개선 단위별로 테스트 실행·커밋.

## 구조적 장치

### 잠금 파일

프로젝트 루트에 한 번에 하나의 잠금 파일만 존재한다:

- `.tdd-phase-red-active`
- `.tdd-phase-green-active`
- `.tdd-phase-refactor-active`

Phase 진입 시 생성 → Phase 종료 시 삭제. `.gitignore`에 추가. skill 최초 실행 시 `.gitignore`에 세 항목이 있는지 확인하고 없으면 추가한다.

### 고아 잠금 확인

skill 시작 시 잠금 파일이 이미 존재하면 이전 작업의 중단 흔적이다. 개발자에게 알린다:
> "이전 TDD 작업의 잠금이 남아 있습니다 (`.tdd-phase-<X>-active`). 어느 Phase를 이어갈지, 아니면 정리하고 새로 시작할지 결정해주세요."

### hook (선택)

공격적 분리가 필요하면 Git pre-commit hook 또는 파일 수정 guard hook을 추가해 Phase 외 파일 수정을 차단한다. 기본 설치는 잠금 파일 + 규약 경고만 제공하고, hook은 팀 결정에 맡긴다.

### Changelog 마커

각 Phase 전환을 Spec §15에 기록한다 (잠금 파일은 `.spec-writing-active`를 별도 사용 — TDD 잠금과 다르다):

```
| 2026-04-20 | TDD Red 완료: AC-05, AC-06, AC-07 테스트 작성 (N개 실패 확인) |
| 2026-04-20 | TDD Green 완료: 위 N개 테스트 통과 |
| 2026-04-20 | TDD Refactor 완료: AuthRepository 추출 |
```

이 기록은 대화 단절 후 어느 Phase에서 이어가야 할지 판단하는 근거가 된다.

## feature 파이프라인과의 연결

feature 파이프라인 4단계에서 방법론으로 **TDD**가 선택된 뒤 5단계(구현)에 진입하면, feature skill은 이 TDD skill의 Phase 구조를 따른다. feature의 Spec↔코드 일치 게이트는 TDD와 충돌하지 않는다:

- **Green Phase에서 Spec과 코드가 어긋나는 현상**이 발견되면, Spec을 먼저 개정한 뒤 Red Phase로 복귀해 테스트를 재작성한다. 직접 구현만 바꾸면 테스트가 이전 계약을 그대로 검증하므로 괴리가 남는다.

## 완료 판정

아래가 모두 참이면 TDD 사이클이 종료된 것으로 본다:

- Spec §14의 모든 AC가 대응 테스트로 번역되었다.
- 모든 테스트가 통과한다.
- 마지막 Phase가 잠금 해제되고 커밋되었다.
- Changelog에 최종 전환(Refactor 완료 또는 마지막 Green 완료)이 기록되었다.

이후 feature 파이프라인의 6단계(검증)로 넘어간다.
