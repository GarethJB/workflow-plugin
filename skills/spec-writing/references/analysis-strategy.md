# 기획서 분석 전략

기획서의 형태(Figma, PDF, 워드, 노션, 이미지 등)는 다르지만, 분석의 목적은 하나다: **내용을 추출하여 15개 Spec 섹션에 매핑한다.**

이 문서는 형식별 읽기 방법과, 형식에 무관한 내용 분석 방법을 정의한다.

## 분석 흐름

```
[형식 판별] → [내용 추출] → [내용 분류] → [섹션 매핑] → [품질 평가] → [빈 섹션 목록 반환]
```

| 단계 | 형식 의존 | 설명 |
|---|---|---|
| 형식 판별 | 예 | 파일 확장자, URL, 첨부 형태로 판별 |
| 내용 추출 | 예 | 형식에 맞는 도구로 텍스트/시각 정보를 읽음 |
| 내용 분류 | **아니오** | 추출된 내용을 의미 기준으로 분류 |
| 섹션 매핑 | **아니오** | 분류된 내용을 15개 섹션에 배치 |
| 품질 평가 | **아니오** | 매핑된 내용의 구체성/완성도 판단 |
| 빈 섹션 목록 | **아니오** | 보완 탐색이 필요한 섹션 식별 |

형식에 의존하는 것은 앞의 두 단계뿐이다. 나머지는 어떤 형식이든 동일하게 적용한다.

---

## 1단계: 형식 판별

첨부된 기획서의 형식을 판별한다.

| 형식 | 판별 기준 | 읽기 도구 |
|---|---|---|
| **PDF** | `.pdf` 확장자 | Read 도구 (PDF 모드) |
| **Figma** | `figma.com` URL | Figma MCP (`get_design_context`, `get_screenshot`) |
| **이미지** | `.png`, `.jpg`, `.jpeg`, `.webp` 등 | Read 도구 (이미지 모드) |
| **마크다운** | `.md` 확장자 | Read 도구 |
| **brainstorming 설계 문서** | `docs/superpowers/specs/*-design.md` 경로 | Read 도구 |
| **워드** | `.docx` 확장자 | Bash로 텍스트 추출 또는 Read 시도 |
| **HTML / 노션 내보내기** | `.html` 확장자 | Read 도구 |
| **텍스트** | `.txt` 또는 확장자 없음 | Read 도구 |
| **대화 내 직접 기술** | 파일 첨부 없이 텍스트로 설명 | 추출 불필요 — 바로 내용 분류로 |

형식이 불분명하면 개발자에게 확인한다:
> "첨부하신 파일의 형식을 확인하고 싶습니다. 어떤 형태인가요?"

### 복합 입력

기획서가 단일 파일이 아닐 수 있다:
- Figma URL + PDF 보충 문서
- 이미지 여러 장 + 텍스트 설명
- 노션 링크 + 구두 보충

이 경우 모든 입력을 순서대로 읽고, 내용을 합산하여 분류한다. 입력 간 충돌이 있으면 (예: Figma와 PDF의 흐름이 다름) 개발자에게 확인한다.

---

## 2단계: 내용 추출

형식별로 적절한 도구를 사용하여 내용을 추출한다.

### PDF

```
Read 도구로 읽는다. 페이지가 많으면 (10페이지 이상) pages 파라미터로 나누어 읽는다.
```

추출 대상:
- 텍스트 본문 (기능 설명, 요구사항, 흐름)
- 표 (데이터 구조, 상태 정의, 에러 코드 등)
- 다이어그램이 있으면 텍스트로 설명된 부분을 우선 추출

### Figma

```
Figma MCP의 get_design_context를 호출한다. URL에서 fileKey와 nodeId를 추출한다.
```

추출 대상:
- 화면 구성 (어떤 화면이 있는가)
- UI 흐름 (화면 간 전환)
- 컴포넌트 목록 (버튼, 입력 필드, 상태 표시 등)
- 디자이너 주석 (annotation)
- 코드 연결 정보 (Code Connect)

Figma는 주로 **행위 명세의 정상 흐름**(화면 단위)과 **범위**(어떤 화면이 포함되는지)를 제공한다. 기술 설계나 에러 처리는 거의 없다.

### 이미지

```
Read 도구로 이미지를 읽는다 (멀티모달).
```

추출 대상:
- 화면 캡처 → UI 구성, 레이아웃, 텍스트
- 다이어그램 → 흐름, 상태 전이, 아키텍처
- 표/차트 → 데이터 구조, 기준값

이미지는 정보 밀도가 텍스트보다 낮을 수 있다. 이미지에서 읽어낸 내용에 확신이 없으면 개발자에게 확인한다:
> "이 이미지에서 [X]로 읽었는데, 맞나요?"

### 마크다운 / 텍스트 / HTML

```
Read 도구로 읽는다.
```

구조화된 형식(제목, 목록, 표)이 있으면 그 구조를 활용하여 내용을 분류한다.

### 대화 내 직접 기술

파일 첨부 없이 개발자가 대화에서 직접 설명한 경우. 추출 단계를 건너뛰고 바로 내용 분류로 넘어간다.

---

## 3단계: 내용 분류

추출된 내용을 의미 기준으로 분류한다. 이 단계부터는 형식에 무관하다.

### 분류 기준

추출된 각 정보 단위(문장, 문단, 이미지의 한 영역)를 아래 카테고리로 분류한다:

| 카테고리 | 인식 신호 | 매핑될 Spec 섹션 |
|---|---|---|
| **목적/배경** | "~를 위해", "~가 필요하여", 프로젝트 배경 설명 | 1. 목적 |
| **용어** | 도메인 특화 단어, 약어, 사내 용어 | 2. 용어 정의 |
| **전제/가정** | "~를 전제로", "~가 완료된 상태에서", "~가 존재한다고 가정" | 3. 전제 |
| **외부 시스템** | SDK 이름, API 엔드포인트, 서드파티 서비스 언급 | 4. 외부 의존성 |
| **기능 동작** | "사용자가 ~하면 ~한다", 화면 흐름, 시나리오 | 5. 행위 명세 |
| **실패/예외** | "~할 수 없는 경우", "에러 시", "실패하면" | 6. 에러 처리 |
| **포함/제외** | "이번 범위에 포함", "~는 제외", "~는 다음 스프린트에" | 7. 범위 |
| **제한/규칙** | "~이어야 한다", "최대 ~까지", 보안/규정 언급 | 8. 제약 |
| **속도/용량** | 응답 시간, 처리량, 메모리, 배터리 관련 수치 | 9. 성능 기준 |
| **구조/설계** | 모듈명, 클래스명, 아키텍처 다이어그램, 인터페이스 | 10. 기술 설계 |
| **플랫폼** | "iOS에서는", "Android에서는", OS 버전, 디바이스 | 11. 플랫폼별 차이 |
| **전환/이전** | "기존 ~에서", 마이그레이션, 하위 호환, 데이터 이전 | 12. 마이그레이션 |
| **테스트** | "테스트 시나리오", "검증 방법", QA 관련 | 13. 테스트 전략 |
| **완료 조건** | "~가 되면 완료", 체크리스트, 인수 조건 | 14. 수락 기준 |

> 섹션 15(변경 이력/Changelog)는 분류 대상이 아니다. Changelog는 Spec 생성 시 자동으로 빈 상태로 생성되며, 이후 파이프라인 진행 중 단계별 기록이 추가된다.

분류가 모호한 경우 (예: "결제 실패 시 3회까지 재시도"는 행위 명세인가 에러 처리인가?) — 가장 밀접한 섹션에 배치하되, 관련 섹션에서도 참조한다.

### 분류 불가 내용

기획서의 일부 내용이 15개 섹션 어디에도 맞지 않을 수 있다:
- 프로젝트 일정 → Spec에 포함하지 않음
- 담당자 배정 → Spec에 포함하지 않음
- 비즈니스 KPI → 목적(1)에 요약으로만 반영

이런 내용은 버리지 말고, 개발자에게 알린다:
> "기획서에서 아래 내용은 Spec에 포함하지 않았습니다: [항목]. 포함이 필요하면 말씀해주세요."

## 4단계: 섹션 매핑

분류된 내용을 15개 Spec 섹션에 배치한다.

### 매핑 규칙

- **하나의 기획서 내용 → 여러 섹션:** 가능. 예: "쿠폰은 1회만 사용 가능하다"는 행위 명세(5)와 제약(8) 모두에 해당.
- **여러 기획서 내용 → 하나의 섹션:** 가능. 기획서의 여러 곳에 흩어진 에러 케이스를 에러 처리(6)에 모은다.
- **기획서 원문을 그대로 옮기지 않는다:** Spec의 형식과 용어에 맞게 재작성한다. 기획서가 "사용자 친화적으로 처리"라고 썼으면, Spec에는 구체적 동작을 적는다.

### 매핑 시 변환

기획서의 표현을 Spec에 적합한 형태로 변환한다:

| 기획서 표현 | Spec 변환 |
|---|---|
| "빠르게 응답한다" | "응답 시간 < [N]ms" (수치를 개발자에게 확인) |
| "적절히 에러를 처리한다" | 에러 유형별 구체적 동작 (보완 탐색에서 확인) |
| "사용자에게 안내한다" | 어떤 메시지를, 어떤 위치에, 어떤 형태로 |
| "기존 기능과 동일하게" | 기존 기능의 구체적 동작을 명시 |
| UI 시안 (Figma/이미지) | 화면별 동작 흐름 + 상태별 UI 변화 |

---

## 5단계: 품질 평가

매핑된 내용의 품질을 평가한다.

### 평가 기준

각 섹션에 매핑된 내용에 대해:

| 품질 수준 | 기준 | 예시 |
|---|---|---|
| **충분** | 구체적이고, 검증 가능하며, 모호하지 않음 | "만료된 쿠폰 적용 시 '이 쿠폰은 만료되었습니다' 토스트 표시" |
| **보완 필요** | 있지만 모호하거나 불완전 | "쿠폰 에러 시 에러 메시지 표시" (어떤 에러? 어떤 메시지?) |
| **없음** | 해당 섹션에 매핑된 내용이 전혀 없음 | 기술 설계 — 기획서에 구조 언급 없음 |

**"보완 필요" 내용은 "없음"보다 중요하다.** 없는 건 명확히 물어보면 되지만, 있지만 모호한 것은 개발자가 "이미 기획서에 적혀 있잖아"라고 생각할 수 있어서, 왜 보완이 필요한지 설명해야 한다:
> "기획서에 '에러 시 적절히 처리'라고 되어 있는데, Spec에는 구체적 동작이 필요합니다. 쿠폰 만료 시에는 어떤 메시지를 보여주나요?"

---

## 6단계: 빈 섹션 목록 반환

품질 평가 결과를 종합하여 빈 섹션 목록을 만든다.

- **충분:** 추가 질문 불필요. Spec에 바로 작성.
- **보완 필요:** 보완 탐색에서 구체화 질문. 왜 보완이 필요한지 함께 설명.
- **없음:** 보완 탐색에서 새로 질문.

이 목록을 개발자에게 보여주고, 보완 탐색을 시작한다.

> "기획서에서 아래 섹션을 채웠습니다: [충분한 섹션 목록].
> 아래 섹션은 추가 질문이 필요합니다:
> - 에러 처리: 기획서에 '적절히 처리'로만 되어 있어 구체화 필요
> - 성능 기준: 기획서에 언급 없음
> - 기술 설계: 기획서에 언급 없음 (개발자 영역)
> - ..."

## 형식별 특성 요약

각 형식이 15개 섹션 중 어디를 잘 채우는지의 경향:

| 형식 | 잘 채우는 섹션 | 보통 빈 섹션 |
|---|---|---|
| **brainstorming 설계 문서** | 목적, 범위, 행위 명세(접근 방식 확정), 기술 설계(구조) | 에러 처리 세부사항, 성능 기준, 테스트 전략 |
| **PDF (기획서)** | 목적, 행위 명세(정상 흐름), 범위 | 기술 설계, 성능 기준, 테스트 전략 |
| **Figma** | 행위 명세(UI 흐름), 범위(화면 목록) | 에러 처리, 기술 설계, 성능 기준, 전제 |
| **이미지 (화면 캡처)** | 행위 명세(현재 상태), 범위 | 거의 모든 How 섹션 |
| **마크다운/텍스트** | 형식에 따라 다름 — 구조화되어 있으면 넓게 커버 가능 | 형식에 따라 다름 |
| **대화 내 설명** | 목적, 행위 명세(개요) | 경계 조건, 기술 설계, 성능 기준, 테스트 전략 |

이 표는 경향일 뿐이다. 매번 실제 내용을 기준으로 판단한다.
