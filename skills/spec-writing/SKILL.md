# Spec Writing

기획서 또는 대화형 탐색을 통해 Spec(소스코드의 SOT)을 작성한다.

## 핵심 원칙

- **Spec = SOT.** 이 skill이 생성하는 Spec은 소스코드의 단일 진실 공급원이다. 코드는 Spec을 따르고, Spec과 다르면 코드가 틀리거나 Spec이 먼저 개정되어야 한다.
- **구조는 고정, 깊이는 가변.** 모든 Spec은 동일한 15개 섹션(4그룹)을 가진다. 가벼운 작업이면 각 섹션이 한 줄, 복잡한 작업이면 상세하게. 압축은 가능, 스킵은 불가.
- **기획서는 입력, Spec은 출력.** 기획서가 있으면 이를 분석하여 Spec으로 변환한다. 기획서가 없으면 대화형 탐색으로 처음부터 작성한다. 어느 경로든 최종 산출물은 동일한 Spec 형식이다.
- **Spec 내용 수정의 유일한 경로.** Spec의 내용(섹션 1~14)을 수정하려면 반드시 이 skill을 거쳐야 한다. 직접 편집은 허용하지 않는다. 이것은 Spec의 SOT 권위를 보호하기 위함이다 — 수정이 구조화된 경로를 통해서만 이루어져야 변경 이유가 기록되고, 무분별한 편집으로 Spec이 오염되지 않는다. 라이프사이클 조작(status 전이, Changelog 기록)은 `feature` 파이프라인에서 수행하며, 이 skill의 범위가 아니다.

## 진입 분기

```
[신규 작성?]                    [기존 Spec 수정?]
    ↓                               ↓
[기획서 첨부?]                  [수정 대상 Spec 읽기]
    ↓ 예        ↓ 아니오             ↓
[기획서 분석] [대화형 탐색]     [수정 범위 확인]
    ↓              ↓                ↓
[15개 섹션 매핑]    ↓           [해당 섹션 보완 탐색]
    ↓              ↓                ↓
[빈 섹션 식별]      ↓               ↓
    ↓              ↓                ↓
[보완 탐색] ←──────┘ ←──────────────┘
    ↓
[완성도 검증]
    ↓
[Spec 생성/갱신]
```

신규 작성과 기존 Spec 수정의 두 가지 모드가 있다. 신규 작성은 기획서 첨부 여부에 따라 다시 분기한다. 모든 경로는 "보완 탐색 → 완성도 검증 → Spec 생성/갱신"으로 합류한다.

### 진입 경로 결정

skill이 시작되면 아래 순서로 경로를 결정한다:

1. 개발자가 기존 Spec ID를 언급하거나 "수정", "개정", "변경"을 요청하면 → **경로 C**
2. 기획서 또는 설계 문서가 첨부되어 있으면 → **경로 A**
   - 기획서: PDF, Figma URL, 이미지, 워드, 마크다운 등
   - **brainstorming 설계 문서** (`docs/superpowers/specs/*-design.md`): 경로 A의 최적 입력. 방향과 접근 방식이 이미 확정되어 있으므로 섹션 매핑 정확도가 높다. 기획서보다 기술 설계(10)와 범위(7)가 더 잘 채워져 있는 경우가 많다.
3. 기획서 없이 새 기능을 설명하면 → **경로 B**
4. 불명확하면 묻는다:
   > "새 Spec을 작성할까요, 기존 Spec을 수정할까요?"

### 복수 기획서 배치 처리

여러 기획서가 동시에 제공된 경우, 아래 순서로 처리한다:

1. 각 기획서가 독립적인 기능 단위인지 확인한다. 하나의 기획서가 다른 기획서의 일부를 포함하면 먼저 분리 기준을 개발자에게 확인한다.
2. **Spec을 하나씩 순서대로 생성한다.** 여러 Spec을 동시에 작성하면 섹션 간 참조와 `related_specs` 연결이 부정확해진다.
3. 첫 번째 Spec 생성 후 `related_specs`에 이후 생성될 Spec들의 예정 ID를 기재한다. 이후 Spec 생성 시 같은 방식으로 연결한다.
4. 배치 처리 중 보완 탐색이 필요한 빈 섹션이 많으면, 개발자에게 전체 일정을 안내한다:
   > "기획서 N개를 Spec으로 변환합니다. 각 Spec마다 빈 섹션에 대한 질문이 있을 수 있습니다. 지금 바로 질문할까요, 아니면 일단 기획서 내용만으로 초안을 먼저 생성할까요?"

## 경로 C: 기존 Spec 수정

기존 Spec의 내용(섹션 1~14)을 수정해야 할 때. `feature` 파이프라인 5단계(구현)에서 Spec 개정이 필요할 때도 이 경로를 사용한다.

### C-1. 수정 대상 Spec 읽기

- 수정할 Spec 파일을 읽는다.
- 개발자에게 어떤 섹션을 왜 수정해야 하는지 확인한다.

### C-2. 수정 범위 확인

- 수정할 섹션을 식별한다.
- 수정이 다른 섹션에 영향을 미치는지 확인한다. 예:
  - 행위 명세(5) 수정 → 수락 기준(14)도 갱신 필요할 수 있음
  - 외부 의존성(4) 변경 → 기술 설계(10)에도 반영 필요할 수 있음
- 영향받는 섹션을 개발자에게 안내한다:
  > "행위 명세를 수정하면 수락 기준도 함께 갱신이 필요할 수 있습니다. 함께 수정할까요?"

### C-3. 보완 탐색

- 수정할 섹션에 대해 대화형 탐색으로 새 내용을 수집한다.
- 기존 내용을 보여주고, 무엇을 어떻게 바꿀지 확인한다.
- 영향받는 섹션도 함께 갱신한다.

### C-4. 갱신

- 잠금 파일을 생성한다 (아래 "Spec 편집 잠금" 참조).
- Spec 파일을 수정한다.
- `updated` 날짜를 갱신한다.
- Changelog(15)에 변경 내용과 사유를 기록한다.
- 잠금 파일을 삭제한다.
- 개발자에게 수정 결과를 제시하고 확인을 받는다.
- 커밋한다.

## 경로 A: 기획서 기반 변환 (신규)

기획서(PDF, 워드, 노션 내보내기, 자유 형식 텍스트 등)가 첨부된 경우.

### A-1. 기획서 분석

기획서를 읽고 내용을 파악한다. 기획서의 형식은 자유롭다(PDF, Figma, 이미지, 마크다운 등) — 정해진 템플릿이 아니므로, 내용의 의미를 기준으로 파악한다.

**형식별 읽기 방법과 상세 분석 전략은 `references/analysis-strategy.md`에 정의되어 있다. 분석 전에 반드시 Read 도구로 이 파일을 읽는다.** 이 문서는 형식 판별 → 내용 추출 → 내용 분류 → 섹션 매핑 → 품질 평가 → 빈 섹션 식별의 6단계 분석 흐름을 정의한다.

기획서에는 보통 **What(무엇을)**이 있고 **How(어떻게)**는 없는 경향이 있다:

| 기획서에 보통 있는 것 | 기획서에 보통 없는 것 |
|---|---|
| 목적, 배경 | 기술 설계 (구조, 인터페이스, 흐름) |
| 기능 설명 (정상 흐름) | 에러 처리의 기술적 세부사항 |
| 범위 (포함/제외) | 플랫폼별 차이 |
| 용어 정의 (일부) | 성능 기준 (수치) |
| UI 흐름, 화면 설명 | 마이그레이션 |
| | 테스트 전략 |
| | 수락 기준 (검증 가능한 형태) |

이 경향은 기획서마다 다르므로, 매번 실제 내용을 기준으로 판단한다.

### A-2. 15개 섹션에 매핑

기획서의 내용을 Spec의 15개 섹션에 배치한다:

- 기획서의 "기능 설명"이나 "유저 스토리" → 행위 명세(5)
- 기획서의 "목적"이나 "배경" → 목적(1)
- 기획서의 "범위" → 범위(7)
- 기획서에 사용된 도메인 용어 → 용어 정의(2)

매핑은 기계적이지 않다 — 기획서의 구조가 Spec의 구조와 일치하지 않을 수 있으므로, 의미를 기준으로 배치한다. 하나의 기획서 문단이 여러 Spec 섹션에 걸칠 수 있고, 반대로 여러 기획서 문단이 하나의 Spec 섹션에 모일 수 있다.

### A-3. 빈 섹션 식별

매핑 후 채워지지 않은 섹션을 파악한다. 보통 "실현" 그룹(기술 설계, 플랫폼별 차이, 마이그레이션)과 "보증" 그룹(테스트 전략, 수락 기준)이 비어 있을 가능성이 높다.

빈 섹션 목록을 개발자에게 보여준다:
> "기획서에서 아래 섹션을 채웠습니다: 목적, 용어 정의, 행위 명세(정상 흐름), 범위.
> 아래 섹션은 기획서에 없어서 추가 질문이 필요합니다: 전제, 외부 의존성, 에러 처리(경계 조건), 제약, 성능 기준, 기술 설계, 플랫폼별 차이, 마이그레이션, 테스트 전략, 수락 기준."

그 다음 빈 섹션에 대해 **보완 탐색**으로 넘어간다.

### 기획서 내용의 검증

기획서에서 가져온 내용도 무조건 신뢰하지 않는다. 매핑 시 아래를 확인한다:

- **모호한 기술:** "적절히 처리한다", "빠르게 응답한다" 같은 표현은 Spec에 그대로 옮기지 않는다. 구체화가 필요하면 보완 탐색에서 질문한다.
- **빠진 경계 조건:** 기획서가 정상 흐름만 기술하고 예외를 다루지 않으면, 예외 흐름을 보완 탐색에서 확인한다.
- **용어 불일치:** 기획서가 같은 개념을 다른 이름으로 부르면, 용어 정의(2)에서 통일한다.

## 경로 B: 대화형 탐색 (처음부터)

기획서 없이 개발자가 구두로 기능을 설명하는 경우.

### B-1. 초기 맥락 파악

개발자의 첫 설명에서 아래를 파악한다:
- **목적:** 이 기능이 왜 필요한가
- **도메인:** 어떤 영역의 기능인가 (결제, 인증, UI 등)
- **기술 배경:** 어떤 플랫폼, 어떤 시스템 위에서 동작하는가

### B-2. 예비 매핑

파악된 정보를 15개 섹션에 예비 매핑한다. 설명만으로 채워진 섹션과 아직 비어 있는 섹션을 구분한다.

### B-3. 보완 탐색

아래 "대화 전략"에 따라 빈/모호한 섹션을 채운다. 이미 알고 있는 것을 확인하지 말고, 모르는 것을 찾아라. 한 번에 하나씩 질문한다.

탐색 종료 조건은 "대화 전략 > 탐색 종료 판단"을 따른다.

### B-4. 완성도 검증 및 Spec 생성

완성도 체크리스트를 확인하고 Spec을 생성한다. 잠금 절차는 "Spec 편집 잠금"을 따른다.

## 대화 전략

기획서 기반이든 처음부터든, 빈 섹션을 채우는 "보완 탐색" 단계에서 아래 전략을 따른다.

### 질문 순서 판단

개발자의 첫 설명(또는 기획서)에서 가장 많은 정보가 나온다. 이를 15개 섹션에 대입해보고, **가장 빈 곳이 아니라 가장 모호한 곳**부터 질문한다.

- 개발자가 "결제에 쿠폰 적용 기능을 만들어야 해"라고 말했다면:
  - 목적(1)과 범위(7)는 어느 정도 파악됨
  - 행위 명세(5)가 가장 모호함 — "쿠폰 적용"이 구체적으로 어떤 동작인지
  - 여기서부터 시작: "쿠폰을 적용한다는 건, 주문 금액에서 할인을 차감하는 건가요, 아니면 특정 상품에만 적용되는 건가요?"

- 기획서에서 정상 흐름은 상세하지만 에러 처리가 없다면:
  - "쿠폰이 만료된 상태에서 적용을 시도하면 어떻게 되어야 하나요?"

**원칙: 이미 알고 있는 것을 확인하지 말고, 모르는 것을 찾아라.**

### 좋은 질문 vs 나쁜 질문

**좋은 질문** — 구체적이고, Spec의 특정 빈 칸을 채우며, 맥락에서 자연스럽게 이어짐:
- "쿠폰이 만료된 상태에서 적용을 시도하면 어떻게 되어야 하나요?" (에러 처리 → 구체적 시나리오)
- "iOS와 Android에서 동작이 다른 부분이 있나요?" (플랫폼별 차이 → 직접적)
- "이 기능의 응답 시간에 대한 기대치가 있나요?" (성능 기준 → 있으면 상세, 없으면 "해당 없음")

**나쁜 질문** — 추상적이거나, 개발자가 답하기 어렵거나, 이미 답을 알고 있는 것:
- "이 기능의 제약 사항은 무엇인가요?" (너무 넓음 — 어떤 제약을 말하는지 좁혀야 함)
- "에러 처리 전략을 설명해주세요." (전략 전체를 한 번에 요청 — 구체적 시나리오별로 나눠야 함)
- "이 기능의 목적이 무엇인가요?" (기획서에 이미 적혀 있는 걸 다시 묻기)

### 모호한 답변 다루기

개발자가 "그건 나중에 결정할게"나 "대충 알아서"라고 답하면:

1. **한 단계만 구체화를 시도한다:** "현재로서는 어떤 방향이 가장 유력한가요? 확정이 아니어도 됩니다."
2. **그래도 모호하면 "미정 — [이유] — 보완 조건: [trigger 또는 기한]" 형식으로 기록하고 넘어간다.** 보완 조건은 "해당 미정이 언제/어떻게 해소되는가"를 명시한다. 예: `미정 — 아직 성능 예산이 정해지지 않음 — 보완 조건: feature 3단계 진입 시`, `미정 — OS 지원 정책 합의 전 — 보완 조건: 2026-05-15까지`. 날짜 예측이 불가능하면 trigger(사건 기반)를 우선한다. 조건 없이 미정만 두는 것은 금지다 — 그러면 Spec이 박물관화된다.
3. **기술 설계에서의 결정 주체 (완화된 규칙).** 섹션 10은 하위 단계별로 규칙이 다르다. 분류 기준은 **프로젝트 고유 맥락 의존도** — 기존 모듈 구성·팀 컨벤션·레거시에 얼마나 종속되는가:
   - **10.1 구조:** 모듈 분할·의존 그래프·아키텍처 패턴 선택은 **개발자가 직접 기술**한다. 이것은 프로젝트 고유 맥락(기존 모듈 위치, 공유 패키지 규약, 팀 합의 아키텍처, 레거시 호환)에 가장 강하게 의존하고, 잘못 잡으면 비가역적 비용이 크기 때문이다. AI는 "어떤 구조를 생각하고 계신가요?" 같은 개방형 질문만 던진다.
   - **10.2 인터페이스 / 10.3 흐름:** AI가 **복수 안(2안 이상) 제안**을 할 수 있다. 단일 안 단정은 금지. 구조가 확정되면 인터페이스·흐름은 도메인 모델·요구사항에서 상당 부분 도출 가능하기 때문이다. 제안 내용에는 `[AI 제안]` 태그를 부착하고, 개발자가 선택·수정한 뒤 태그를 제거한다. 2안을 낼 수 없는 경우에는 단일 안 + 트레이드오프 서술로 대체할 수 있다.
   - 이것이 원래 원칙("대신 결정하지 않는다")의 의도를 가장 큰 비가역 비용 지점(구조)에서 엄격히 보존하면서도, 도출 가능한 영역에서 AI의 생산성을 활용하는 방법이다.

## 깊이 판단 기준

"가변 깊이"를 적용하려면 작업의 무게를 판단해야 한다. 아래 신호들을 참고한다:

| 신호 | 가벼움 | 무거움 |
|---|---|---|
| 영향 범위 | 단일 화면 / 단일 파일 | 다수 모듈 / 크로스 모듈 |
| 상태 변경 | 없음 또는 로컬 | DB, 서버 상태, 사용자 데이터 |
| 외부 연동 | 없음 | API, SDK, 서드파티 서비스 |
| 비즈니스 민감도 | 낮음 (UI 텍스트 수정 등) | 높음 (결제, 인증, 개인정보) |
| 플랫폼 차이 | 없음 | iOS/Android 구현이 다름 |
| 마이그레이션 | 없음 | 기존 데이터/코드 전환 필요 |

**가벼운 작업 예시 — "설정 화면에 다크모드 토글 추가":**
- 목적: 한 문장
- 행위 명세: 토글 ON/OFF 동작 한 줄씩
- 기술 설계: "UserDefaults에 Bool 저장" 한 줄
- 성능 기준: "해당 없음"
- 전체 Spec: 반 페이지

**무거운 작업 예시 — "결제 모듈에 쿠폰 적용 기능 추가":**
- 목적: 비즈니스 맥락 포함 3-4줄
- 행위 명세: 정상 흐름 + 경계 조건 (만료, 최소 금액, 중복 적용 등) 상세
- 에러 처리: 에러 유형별 테이블
- 기술 설계: 모듈 구조, 인터페이스, 시퀀스 다이어그램
- 성능 기준: 응답 시간 < 200ms
- 전체 Spec: 3-5 페이지

깊이를 초기 대화(또는 기획서의 분량)에서 먼저 가늠하고, 질문의 상세 수준을 그에 맞춘다.

## Spec 형식

Spec은 `templates/spec-template.md`의 구조를 정확히 따른다. **Spec을 생성하기 직전에 Read 도구로 이 파일을 읽어 실제 템플릿 구조를 확인한다.** 아래는 그룹별 역할과 존재 이유다.

### 그룹 구조와 역할

**정의 (Definition) — 섹션 1~4:**
이 Spec이 다루는 세계를 정의한다. 목적, 용어, 전제, 외부 의존성. 여기에 있는 전제가 깨지면 Spec 전체를 재검토해야 하므로, 이 그룹은 가장 먼저 작성되어야 할 가능성이 높다 (반드시 그런 건 아님 — 맥락에 따라 다름). 기획서가 있으면 이 그룹은 대부분 채워진다.

섹션 1(목적)은 **"누가 왜"를 함께** 기록한다. 가급적 `[사용자 유형]이 [목적]을 달성하기 위해 [기능의 역할]을 한다` 형식(Job Story)을 포함한다. 예: "재방문 사용자가 빠른 진입을 하기 위해 자동 로그인을 제공한다." 또한 **성공 지표(measurable outcome)**를 한두 줄로 덧붙일 수 있다 — 전환율·DAU·P99 도달 시간 같은 비즈니스/UX 지표. 기술 지표는 섹션 9로 분리한다.

**계약 (Contract) — 섹션 5~9:**
코드가 반드시 만족해야 할 조건. 코드 리뷰 시 이 그룹만 대조하면 Spec 준수 여부를 판단할 수 있다. 이 그룹이 Spec의 핵심이다. 기획서는 행위 명세(5)의 정상 흐름을 제공하지만, 경계 조건·에러 처리·성능 기준은 보완 탐색이 필요한 경우가 많다.

섹션 9(성능 기준)는 표에 **"유형" 컬럼**을 포함한다. 표준값은 `기술`(응답 시간, 메모리, 배터리 등 엔지니어링 지표) 또는 `비즈니스`(전환율, 이탈률 등 제품 지표). 두 유형을 한 표에 섞어 관리하되, 비즈니스 지표의 원 출처(섹션 1 성공 지표)와 일관되게 기재한다.

**실현 (Realization) — 섹션 10~12:**
계약을 이행하는 구체적 방법. 코드의 구조가 여기와 다르면 Spec을 개정해야 한다. 이 그룹의 기술 설계(10)에서 AI와 개발자의 역할 분담은 **"모호한 답변 다루기"의 3번 규칙**을 따른다. 요약: 10.1(구조)은 개발자가 직접 기술한다 — 프로젝트 고유 맥락 의존도가 가장 높고 비가역 비용이 크기 때문이다. 10.2(인터페이스)·10.3(흐름)은 AI가 복수 안을 제안할 수 있고 `[AI 제안]` 태그로 표시한다. AI는 구조 섹션에 대해 "어떤 구조를 생각하고 계신가요?"처럼 개방형 질문만 사용한다. 기획서에는 이 그룹이 거의 없다.

**보증 (Assurance) — 섹션 13~15:**
계약 이행을 증명하는 방법. 수락 기준 전항 통과가 Spec의 shipped 전이 게이트다. 기획서에는 이 그룹이 거의 없다.

### 프로젝트 전역 맥락은 CLAUDE.md

프로젝트 전반에 걸쳐 반복되는 아키텍처 규칙·API 카탈로그·상태 관리 패턴·스타일 컨벤션은 기능별 Spec에 반복 기술하지 않는다. **Claude Code는 모든 세션에 `CLAUDE.md`를 자동 주입**하므로, 이 파일이 프로젝트 전역 맥락 문서(living handbook) 역할을 수행한다. 일반 Spec의 §4·§10.1은 CLAUDE.md에 이미 명시된 규약을 **반복하지 않고**, 해당 Spec 고유의 의존성·구조 결정만 기록한다.

패턴 환류 기록: `feature` 파이프라인 7단계(완료)에서 재사용 가능한 새 패턴·컨벤션이 발견되면 해당 Spec Changelog에 `pattern-extracted: [패턴명]` 마커만 남긴다. CLAUDE.md는 SOT 중 하나이므로 **파이프라인이 직접 수정하지 않는다** — 마커는 향후 사람이 CLAUDE.md 개정 여부를 검토할 때의 원천 자료로 쓰인다.

### 에러 처리 표준값

에러 처리(섹션 6) 테이블 작성 시 아래 표준값을 사용한다. 자유 텍스트 대신 이 목록에서 선택하여 Spec 간 일관성을 유지한다.

**복구 전략**

| 값 | 의미 |
|---|---|
| `버튼 재활성화` | API 호출 중 비활성화된 버튼을 다시 활성화 |
| `화면 이동: [대상]` | 지정 화면으로 이동 (예: `화면 이동: 로그인`) |
| `재시도: [N]회` | 자동 재시도 (예: `재시도: 3회`) |
| `무시` | 별도 처리 없이 계속 진행 |

**사용자 피드백**

| 값 | 의미 |
|---|---|
| `토스트` | 하단 토스트 메시지 |
| `다이얼로그` | 확인/취소 다이얼로그 |
| `인라인 에러` | 입력 필드 하단 에러 텍스트 |
| `없음` | 사용자에게 피드백 없음 |

**로깅 기준**

| 값 | 의미 |
|---|---|
| `error` | 에러 로그 (즉시 확인 필요) |
| `warn` | 경고 로그 (추적 필요) |
| `info` | 정보 로그 |
| `없음` | 로깅 안 함 |

보완 탐색 중 에러 처리를 물을 때는 에러 시나리오별로 위 선택지를 제시한다:
> "영양소 계산 API가 실패하면 어떻게 처리할까요? 복구 전략: 버튼 재활성화 / 화면 이동 / 재시도 / 무시"

### 제약 탐색 가이드

제약(섹션 8)은 "제약 사항이 무엇인가요?" 같은 광범위한 질문으로는 채울 수 없다. 아래 카테고리를 기준으로 기능의 성격에 맞는 항목만 골라 구체적으로 묻는다.

| 카테고리 | 탐색 질문 | 해당할 가능성이 높은 경우 |
|---|---|---|
| **보안** | "인증/권한이 필요한가? 민감 데이터(개인정보, 결제, 토큰)를 다루는가?" | 로그인, 결제, 개인정보 입력 |
| **접근성** | "스크린 리더 지원이 필요한가? 최소 터치 영역 등 접근성 기준이 있는가?" | 모든 UI 기능 |
| **호환성** | "최소 OS 버전 요구사항이 있는가? 특정 기기나 화면 크기 제한이 있는가?" | 플랫폼 API 사용, 신규 기능 |
| **정책/규정** | "앱스토어 정책, 법적 요구사항, 사내 보안 정책을 따라야 하는가?" | 결제, 개인정보 수집, 구독 |

기획서에서 제약이 명시되지 않은 경우, 위 카테고리 중 기능 성격에 맞는 것만 선별하여 질문한다. 모든 카테고리를 항상 물을 필요는 없다.

### 수락 기준 문장 형식

수락 기준(섹션 14)의 각 항목은 반드시 아래 형식을 따른다:

> **"AC-NN: [조건] 시 [관찰 가능한 결과]한다"**

- **AC 번호 강제:** 각 항목은 `AC-01`, `AC-02`…처럼 두 자리 번호로 시작한다. 번호는 테스트·PR·버그 리포트와의 교차 참조에 사용된다. 섹션 내에서 번호가 중복되지 않도록 한다.
- **결과의 관찰 가능성:** 결과는 코드 없이 외부에서 확인 가능해야 한다. 형식을 벗어난 항목은 재작성을 요청한다.
- **선택 컬럼 — 우선순위(P1/P2):** MVP 스코핑이 필요한 큰 기능은 각 AC 뒤에 `[P1]` 또는 `[P2]` 라벨을 부착할 수 있다. 라벨이 없으면 모두 동일 우선순위로 간주한다. 작은 기능은 생략해도 된다.

**좋은 예:**
- `AC-01: 활동량 미선택 시 다음 버튼이 비활성화된다`
- `AC-02 [P1]: API 호출 중 시작하기 버튼이 비활성화된다`
- `AC-03 [P2]: 30초 초과 시 토스트 메시지가 표시되고 버튼이 재활성화된다`

**나쁜 예 → 처리 방법:**

| 나쁜 예 | 문제 | 처리 |
|---|---|---|
| `UI가 자연스럽다` | 관찰 불가 | 구체적 동작으로 재작성 요청 |
| `에러를 잘 처리한다` | 조건 없음 | 에러 유형별로 분리하여 재작성 |
| `빠르게 응답한다` | 수치 없음 | 성능 기준(섹션 9)으로 이동 |
| `TDEE를 계산한다` | 결과가 관찰 불가 | "올바른 TDEE 값이 결과 화면에 표시된다"로 재작성 |

보완 탐색 중 수락 기준을 채울 때는 이 형식으로 유도한다:
> "이 기능이 완료됐다고 판단하는 기준을 '[조건] 시 [결과]한다' 형식으로 알려주세요."

- `id`: SPEC- 접두어 + 순번. `specs/SPEC-*.md`를 확인하여 가장 큰 번호 다음을 사용한다. 파일이 없으면 `SPEC-001`부터 시작한다. 번호에 gap이 있어도 가장 큰 번호 기준으로 증가한다 (예: SPEC-001, SPEC-003 존재 → 다음은 SPEC-004).
- `author`: `git config user.name`을 실행하여 사용한다. 확인이 어려우면 개발자에게 직접 묻는다.
- `status`: 이 skill이 생성하는 Spec은 항상 `draft`
- `created`/`updated`: 생성 당일 날짜
- `related_specs`: 아래 기준 중 하나 이상에 해당하는 Spec의 ID를 목록에 추가한다. 해당 Spec이 없으면 빈 배열(`[]`)로 둔다.
  - 같은 API 엔드포인트를 공유하는 Spec
  - 화면 진입/출구가 직접 연결된 Spec (예: 로그인 → 온보딩, 홈 → 식단 등록)
  - 동일 도메인 객체(식단, 음식, 회원 등)를 다루는 연관 Spec

> **Changelog 장기 운영 각주.** 섹션 15(Changelog)는 같은 파일에 누적된다. 장기 운영으로 Changelog가 본문보다 비대해질 경우 팀 합의로 Spec 파일을 분할할 수 있다 (예: `shipped` 전이 시 이전 Changelog를 `CHANGELOG-<timestamp>.md`로 archive). 분할 규칙은 이 skill의 범위가 아니며, 필요 시 `feature` 파이프라인과 함께 개정한다.

## 이 skill이 하는 것

- 기획서가 첨부되면 분석하여 15개 섹션에 매핑한다
- 기획서가 없으면 대화형 탐색으로 처음부터 정보를 수집한다
- 빈 섹션을 식별하고 보완 질문을 던진다 (한 번에 하나씩)
- 모든 섹션이 채워졌는지 검증한다
- Spec 문서(draft 상태)를 생성한다

## 이 skill이 하지 않는 것

- **섹션 순서를 강제하지 않는다.** 1번부터 순서대로 물으면 기계적 인터뷰가 된다. 대화의 흐름에 따라 자연스럽게 섹션을 넘나드는 것이 이 skill의 핵심 가치다.
- **단일 구현 방법을 단정하지 않는다.** 기술 설계(섹션 10)에서 AI가 단일 안을 결정적으로 단정하는 것은 금지다. 그러나 **복수 안 제안과 트레이드오프 서술은 허용**한다. 세부 규칙은 "대화 전략 > 모호한 답변 다루기 > 3번"을 따른다 — 10.1 구조는 개발자가 직접 기술하고(프로젝트 고유 맥락 의존이 가장 높음), 10.2 인터페이스·10.3 흐름은 AI가 `[AI 제안]` 태그와 함께 복수 안을 낼 수 있다. 이 분업이 SOT로서의 Spec 소유권(개발자)과 AI의 생산적 기여를 양립시킨다.
- **Spec을 승인하지 않는다.** 승인은 `feature` skill 파이프라인의 다음 단계(2단계)에서 이루어진다. 이 skill의 역할은 draft 생성까지다.
- **코드를 작성하지 않는다.** 이 skill은 문서를 만든다. 코드는 파이프라인의 5단계(구현)에서 작성된다.
- **기획서를 그대로 복사하지 않는다.** 기획서의 모호한 표현, 빠진 경계 조건, 용어 불일치를 식별하고 보완한다.

## 완성도 체크리스트

Spec 생성 전 아래를 확인한다:

- [ ] 15개 섹션이 모두 존재하는가
- [ ] 각 섹션에 내용이 있는가 (해당 없음이면 "해당 없음" 명시)
- [ ] 용어 정의(Glossary)에 Spec 내 도메인 용어가 모두 정의되어 있는가
- [ ] 행위 명세(Behavior)에 정상 흐름과 경계 조건이 있는가
- [ ] 수락 기준(Acceptance Criteria)이 체크리스트 형식인가
- [ ] 수락 기준의 각 항목이 `AC-NN: [조건] 시 [관찰 가능한 결과]한다` 형식인가 (번호 중복 없음)
- [ ] frontmatter의 id, title, status, author, created, updated가 모두 채워졌는가
- [ ] 기획서에서 가져온 내용에 모호한 표현("적절히", "빠르게" 등)이 남아있지 않은가
- [ ] "미정"으로 남긴 모든 항목이 `미정 — [이유] — 보완 조건: [trigger 또는 기한]` 형식을 따르는가
- [ ] 섹션 10.2·10.3에 `[AI 제안]` 태그가 남아 있다면 개발자 검토/선택이 필요함을 개발자에게 알렸는가 (10.1은 AI 제안 금지)

## Spec 편집 잠금

Spec 파일(`specs/SPEC-*.md`)은 **2계층 가드**로 보호된다. 각 계층은 다른 시나리오를 방어한다.

| 계층 | 방어 시나리오 | 구현 |
|---|---|---|
| **세션 가드** | Claude Code 세션 내에서 AI가 스킬 경로를 우회해 Spec을 편집하려는 시도 | `.spec-writing-active` 잠금 파일 + [hooks/guard-spec.sh](../../hooks/guard-spec.sh) |
| **커밋 가드 (후속 개정 예정)** | IDE 직접 편집, 다른 AI 도구, Claude Code 외부 경로를 통한 커밋 | Git pre-commit hook (별도 플랜으로 구현) |

두 계층은 이중 방어가 아니라 **분업**이다 — 세션 가드는 런타임을, 커밋 가드는 커밋 직전을 본다. 팀 규율을 통한 `--no-verify` 우회 방지는 스킬 범위 밖이다.

**Spec을 생성하거나 수정할 때 반드시 아래 절차를 따른다 (세션 가드 절차):**

0. **고아 확인:** `.spec-writing-active`가 이미 존재하면 이전 작업이 중단된 흔적이다. 개발자에게 알린다: "이전 작업의 잠금 파일이 남아 있습니다. 정리하고 진행합니다." `rm .spec-writing-active`로 삭제한 뒤 1번부터 시작한다.
1. 잠금 파일 생성: `touch .spec-writing-active`
2. Spec 파일 생성/수정 (Edit 또는 Write 도구)
3. 잠금 파일 삭제: `rm .spec-writing-active`

잠금 파일을 삭제하지 않고 남겨두면 다른 경로에서도 Spec을 편집할 수 있게 되므로, 반드시 작업 후 즉시 삭제한다. 잠금 파일은 `.gitignore`에 포함하여 커밋하지 않는다. **skill 최초 실행 시 `.gitignore`에 `.spec-writing-active` 항목이 있는지 확인하고, 없으면 추가한다.**
