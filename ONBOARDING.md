# Native Dev Team에 오신 것을 환영합니다

## 플러그인 한눈에 보기

**`workflow-plugin`은 네이티브 개발부서의 워크플로우를 Spec 중심으로 구조화하는 Claude Code 플러그인이다.** 기획·합의·구현·검증의 경계를 느슨한 대화 대신 *파일로 정착된 계약*으로 바꾸고, Claude가 그 계약을 강제·참조·갱신하도록 만든다.

### 워크플로우 요약

```
[기획 문서·아이디어]
        │
        ▼
  ┌───────────────┐   ┌─────────────────────────────┐
  │ spec-writing  │   │ feature (7단계)               │
  │ (Spec 작성)   │──▶│ Spec → 승인 → 계획 → Practice  │
  └───────────────┘   │       → 구현 → 검증 → 완료     │
        ▲             └────────────┬────────────────┘
        │                          │ 검증 실패
        │ 회귀 테스트 추가           ▼
        │             ┌─────────────────────────────┐
        │             │ debug (8단계)                │
        └─────────────│ 증상 → Spec 확인 → 가설 → 검증 │
                      │     → 원인 → 수정 → 회귀 → 완료│
                      └─────────────────────────────┘
```

- **진입점은 항상 Spec.** 새 기능이든 버그든 먼저 Spec에서 정의·수락 기준을 확인하거나 갱신한다.
- **Practice**는 위험도 3단계에 맞춰 3종으로 운영한다 — **고위험: TDD**, **중위험: Walking Skeleton**, **저위험: Implementation First**. `feature` 4단계에서 선택한다.
- **`feature`와 `debug`는 양방향으로 묶여 있다.** 구현 검증 실패는 debug로, debug에서 드러난 Spec 결함(`spec_flags`)은 다음 feature 계획에 반영된다.

### 핵심 원칙

1. **Spec이 SOT(Single Source of Truth)다.** Spec·코드·대화가 엇갈리면 Spec이 기준. 코드가 먼저 달라지면 Spec을 갱신하고, Spec이 잘못됐으면 코드는 건드리기 전에 Spec부터 고친다.
2. **15섹션 고정 구조.** Spec은 4그룹(정의 §1-4 / 계약 §5-9 / 실현 §10-12 / 보증 §13-15)으로 고정된다. 섹션 추가·삭제 없이 형식을 유지해 검색·참조·리뷰 비용을 낮춘다.
3. **Spec 파일은 `spec-writing` 스킬로만 편집.** 훅 + 락 파일로 직접 편집을 차단해, 대화 맥락 없이 수정되는 것을 막는다.
4. **수락 기준은 번호로 말한다.** `AC-NN` 번호와 `§N` 표기를 커밋·이슈·테스트명에서 그대로 쓴다. "어디가 왜 실패했는가"가 검색 가능해진다.
5. **AI는 대신 결정하지 않는다. 단, 복수 안은 제안한다.** §10.1(구조)은 개발자 전담, §10.2(인터페이스)·§10.3(흐름)은 AI가 2개 이상의 안을 `[AI 제안]` 태그로 제시하고 개발자가 선택·수정한다.
6. **단계마다 Changelog에 흔적을 남긴다.** 파이프라인 중간에서 중단되더라도 재개 지점을 되찾을 수 있도록, 각 단계 완료 시 §15에 마커를 기록한다.
7. **계획이 먼저, 구현은 나중.** 불확실성이 큰 작업은 Plan 모드·Spec 초안으로 먼저 합의를 만들고, 그 위에서 코드가 쓰인다.
8. **프로젝트 전역 맥락은 CLAUDE.md.** 아키텍처 규칙·API 카탈로그·상태 관리·스타일 컨벤션은 CLAUDE.md에 있다. Claude가 모든 세션에서 자동 로드하므로 일반 Spec은 이를 반복 기술하지 않는다.
9. **CLAUDE.md도 SOT다.** `feature`·`debug` 파이프라인은 CLAUDE.md를 직접 수정하지 않는다. 새 패턴이 발견되면 Spec Changelog에 `pattern-extracted:` 마커만 남기고, 반영 여부는 사람이 별도 리뷰에서 결정한다.
10. **요청은 구조화된 5필드로 던진다.** 문제·맥락(Spec·AC·CLAUDE.md 제약)·기대 결과·제약·산출물 형식을 명시한다 — 모호한 요청은 스킬도 구제 못 한다. 상세는 [docs/structured-request.md](docs/structured-request.md).

### 알아둘 스킬·커맨드
- [/onboard](commands/onboard.md) — 플러그인 자체 온보딩. `workflow-plugin`이 적용된 레포에서 실행하면 스킬 목록·Spec 규약·진입점을 빠르게 안내합니다.
- `spec-writing` 스킬 — Spec 작성·수정 전담 (15섹션 고정, 락 파일로 외부 편집 차단). "spec 작성해줘", "spec 수정해줘"로 트리거.
- `feature` 스킬 — 7단계 기능 개발 파이프라인 (Spec 작성 → 승인 → 계획 → Practice 선택 → 구현 → 검증 → 완료). "feature 스킬로 시작해줘"로 트리거.
- `debug` 스킬 — 8단계 버그 수정 파이프라인 (증상 기록 → Spec 확인 → 가설 → 검증 → 원인 확정 → 수정 → 회귀 → 완료). "debug 스킬로 시작해줘"로 트리거.
- `practice-*` 스킬 (tdd, walking-skeleton, implementation-first) — `feature` 4단계에서 위험도에 맞춰 선택됩니다. 고위험은 TDD, 중위험(통합 리스크)은 Walking Skeleton, 저위험은 Implementation First가 기본값.
- `/compact` — 내장 커맨드. 대화가 길어지면 이전 내역을 요약·압축합니다.

## 팀 팁

- **락 파일(`.spec-writing-active` 등)이 남아 있으면 먼저 확인.** 다른 세션이 작업 중일 수 있다. 본인이 만든 게 아니라면 팀 채널에서 먼저 물어보고, 고아 락이 확실하면 제거 후 재시도.
- **새 기능은 `feature`, 버그는 `debug`로 시작한다.** 단발 질문이면 스킬 호출 없이 대화로 진행해도 되지만, 코드에 손이 닿는 작업은 파이프라인 진입이 원칙.
- **Practice 선택 기본값.** 결제·인증·데이터 정합성 같은 **고위험**은 TDD, 새 SDK·프로토콜·통신 경로 개척 같은 **중위험**(통합 리스크)은 Walking Skeleton, 내부 도구·프로토타입 같은 **저위험**은 Implementation First.
- **CLAUDE.md를 먼저 읽는다.** 프로젝트 전역 규약(아키텍처·API 패턴·컨벤션)이 여기 있다. 새 Spec 작성 전에 훑어보면 기능 Spec과 전역 규약의 충돌을 예방할 수 있다.
- **구조화된 요청 5필드.** 요청은 "문제 / 맥락(Spec·AC·CLAUDE.md 제약) / 기대 결과 / 제약 / 산출물 형식"을 명시. 템플릿과 좋은 예는 `docs/structured-request.md` 참고.
- **패턴 환류 기록.** feature 7단계 완료 시 재사용 가능한 패턴이 발견되면 Changelog에 `pattern-extracted:` 마커를 남긴다. 파이프라인은 CLAUDE.md를 건드리지 않으며, 반영 여부는 팀의 별도 리뷰에서 결정한다 — 마커가 그 리뷰의 원천 자료가 된다.

## 시작 작업

처음이라면 이 순서로 30분 정도 투자하는 걸 권장합니다.

1. **`/onboard` 실행.** 이 레포 루트에서 슬래시 커맨드로 호출하면 스킬 구성·Spec 규약·진입점을 요약해준다.
2. **CLAUDE.md 읽기.** 프로젝트 전역 규약이 여기 있다. Claude가 세션마다 자동 로드하지만, 사람이 먼저 맥락을 알아두면 기능 Spec이 전역 규약과 충돌하는 결정을 방지할 수 있다.
3. **최신 기능 Spec 1건 읽기.** `specs/005-splash-login-flow/spec.md`를 처음부터 끝까지. 15섹션 구조가 실제로 어떻게 채워지는지 감이 온다.
4. **`skills/feature/SKILL.md`와 `skills/spec-writing/SKILL.md` 훑기.** 각각 7단계·15섹션의 WHY가 본문에 적혀 있다. 외울 필요는 없고 "무엇이 어디에 적혀 있는지"만 위치를 기억.
5. **`docs/structured-request.md` 훑기.** 요청 5필드 템플릿. 이후 AI에게 말을 걸 때 머릿속으로 이 형식을 따라보면 사이클이 짧아진다.
6. **작은 Spec 하나 실습.** 간단한 유틸리티를 골라 `spec-writing` 스킬로 Spec 초안을 써본다. 실제 머지는 하지 않아도 되고, §10 기술 설계에서 AI가 복수 안을 제안할 때 어떻게 선택하는지를 직접 경험하는 게 목표.
