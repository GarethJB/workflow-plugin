# /onboard

사용자가 `workflow-plugin`의 구성을 파악하도록 읽기 전용 안내를 제공한다. 파일 생성/수정은 금지한다.

## 실행 순서

### 1. 맥락 감지 (조용히, 읽기 전용)

다음 순서로 현재 작업 디렉토리를 판별한다:

1. `.claude-plugin/plugin.json`을 읽어 `"name": "workflow-plugin"`이 있으면 → **플러그인 개발자 분기**
2. `specs/` 디렉토리가 있으면 → **레포 첫 진입 팀원 분기**
3. 둘 다 아니면 → **플러그인 설치 개인 분기**

감지 과정은 사용자에게 노출하지 말고 결과만 안내에 반영한다.

### 2. 공통 소개 출력 (모든 분기 공통)

다음 내용을 간결하게 한 번에 출력한다. 50줄 이내.

```
# workflow-plugin

네이티브 개발부서의 워크플로우를 Spec 중심으로 구조화하는 Claude Code 플러그인.
Spec이 SOT(Single Source of Truth)이며, 코드와 Spec의 정렬을 강제한다.

## 스킬 구성

**최상위 (3)** — 파이프라인 진입점
- `spec-writing` : Spec 작성·수정 (15섹션 고정, 락 파일 보호)
- `feature` : 새 기능 개발 7단계 파이프라인 (Spec 작성 → 승인 → 계획 → Practice 선택 → 구현 → 검증 → 완료)
- `debug` : 버그 수정 8단계 파이프라인 (증상 → Spec 확인 → 가설 → 검증 → 원인 → 수정 → 회귀 → 완료)

**Practice (3)** — `feature` 4단계에서 위험도별로 선택
- `practice-tdd` (고위험) / `practice-walking-skeleton` (중위험) / `practice-implementation-first` (저위험)

> 위 개수는 예시. 실제 구성은 `skills/` 디렉토리로 확인.

## Spec 규약 (요약)

- 15섹션 고정 구조, 4그룹 (정의 §1-4 / 계약 §5-9 / 실현 §10-12 / 보증 §13-15)
- 참조 표기: `§N` (섹션), `AC-NN` (수락 기준)
- Spec 파일은 `spec-writing` 스킬을 통해서만 편집 (락 파일 + 훅으로 보호)

## 시작 진입점

- 새 기능: "feature 스킬로 시작해줘"
- 버그 수정: "debug 스킬로 시작해줘"
- Spec만 작성: "spec-writing 스킬로 시작해줘"
```

### 3. 분기별 추가 한 줄 (맥락에 맞춰)

#### 플러그인 설치 개인 분기

다음 한 줄 덧붙임:

> 현재 디렉토리에는 아직 `specs/`가 없다. 이 플러그인은 레포 단위 워크플로우용이니, 실제 프로젝트 디렉토리에서 `feature 스킬로 시작해줘`라고 말하면 진입한다.

#### 레포 첫 진입 팀원 분기

다음을 수행:

1. `specs/` 하위 디렉토리 개수를 센다 (`ls specs/`).
2. 가장 최근 Spec 1건의 `spec.md`에서 frontmatter `title`을 읽는다.
3. 루트의 `CLAUDE.md` 존재 여부를 확인한다.
4. 루트에 `.spec-writing-active` 파일이 있는지 확인한다.
5. 다음 형식으로 출력:

> 이 레포에는 **N개의 Spec**이 있음. 가장 최근: `SPEC-XXX — 제목`.
> (CLAUDE.md 존재 시) 프로젝트 전역 규약은 `CLAUDE.md`에 있으니 먼저 훑어본다 — Claude가 세션마다 자동 로드하지만 사람이 먼저 맥락을 알아두는 편이 안전하다.
> 15섹션 구조가 궁금하면 기능 Spec을 열어본다. 요청은 `docs/structured-request.md`의 5필드를 따른다.
> (락 파일 발견 시 추가) ⚠️ `.spec-writing-active` 락 파일 존재 — 다른 세션이 Spec을 편집 중일 수 있음.

#### 플러그인 개발자 분기

다음 한 줄 덧붙임:

> 이 저장소는 플러그인 자체 소스다. 스킬을 수정하려면 `CLAUDE.md`와 각 `skills/*/SKILL.md`를 참고하고, 최근 변경 내역은 `git log`로 확인.

### 4. 종료 문구

마지막에 한 줄:

> 지금은 이 안내가 전부다. 특정 스킬을 더 알고 싶으면 그 이름(예: "feature 스킬 설명해줘")을 말하면 해당 스킬이 트리거된다.

## 제약

- 파일을 생성하거나 수정하지 않는다. `Read`, `Glob`, `Bash`(ls) 등 읽기 전용 도구만 사용한다.
- 출력은 50줄 이내로 유지한다. 길어지면 요약한다.
- Spec 내용을 인용할 때도 `title`만 사용. 본문 발췌 금지.
