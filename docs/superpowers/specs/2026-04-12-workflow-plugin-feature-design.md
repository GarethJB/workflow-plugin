# workflow-plugin `feature` 커맨드 설계

> 상태: 승인됨 (2026-04-12)

## 1. 플러그인의 정체성

### 목적

네이티브 개발부서의 워크플로우를 정립한다.

### 핵심 원칙

- **Spec = SOT.** Spec이 소스코드의 단일 진실 공급원(Source of Truth)이다. 코드는 Spec을 따르고, Spec과 다르면 코드가 틀리거나 Spec이 먼저 개정되어야 한다.
- **구조적 조형.** 플러그인은 워크플로우를 안내하거나 추천하는 것이 아니라, 구조 자체를 강제한다. 단계를 선택하거나 건너뛸 수 없다.
- **고정 구조, 가변 깊이.** 모든 작업이 동일한 파이프라인을 통과하되, 각 단계의 밀도가 작업 규모에 따라 달라진다. 압축은 가능, 스킵은 불가.

### 적용 방법론

| 방법론 | 적용 방식 |
|---|---|
| SDD (Spec-Driven Development) | 파이프라인의 기본 기조. 모든 작업이 Spec 작성으로 시작 |
| Trunk-Based + PR | 가벼운 UI 변경 시 Spec 각 섹션이 축약형으로 흐름 (별도 경로가 아닌, 동일 구조의 압축) |
| TDD | 구현 직전 판단 게이트에서 플러그인이 제안, 사용자가 판단. 구현 정합성을 높이기 위해 작업 성격에 따라 고려 |

방법론은 사용자가 선택하는 옵션이 아니라, 구조에 녹아 있는 재료다. TDD만 유일하게 "제안 + 사용자 판단" 위치에 놓인다. 이는 자동 분류의 경계 분쟁을 피하면서도, 판단해야 할 순간을 구조적으로 보장하기 위함이다.

## 2. Spec 중심 파이프라인

### 단계 흐름

```
[Spec 작성] → [Spec 승인] → [계획] → [TDD 판단 게이트] → [구현] → [검증] → [완료]
```

### 각 단계 정의

| 단계 | Spec 상태 | 행위 | 산출물 |
|---|---|---|---|
| **Spec 작성** | `draft` | `spec-writing` skill이 대화형 탐색으로 Spec 문서를 생성 | spec.md (draft) |
| **Spec 승인** | `draft` → `approved` | 작성된 Spec을 리뷰 후 승인 | spec.md (approved) |
| **계획** | `approved` | Spec을 기반으로 구현 계획 수립 (파일, 모듈, 작업 분해) | plan.md |
| **TDD 판단 게이트** | `approved` | "이 작업에 TDD를 적용할 것인가?" 플러그인이 제안, 사용자가 판단. 결과를 Spec에 기록 | spec.md에 TDD 판단 기록 |
| **구현** | `approved` → `implementing` | TDD 선택 시 테스트 우선 사이클, 미선택 시 일반 구현. Spec ↔ 코드 일치 게이트 존재 | 소스 코드, 테스트 코드 |
| **검증** | `implementing` | 수락 기준 체크리스트 통과 확인 | 검증 결과 |
| **완료** | `implementing` → `shipped` | Spec status를 shipped로 전이 | spec.md (shipped) |

### Spec ↔ 코드 일치 게이트

구현 단계에서 코드가 Spec과 다르게 흘러갈 경우:
- 코드를 Spec에 맞게 수정하거나
- Spec을 먼저 개정한 뒤 코드를 계속 작성

Spec과 코드가 불일치한 상태로 검증 단계에 진입할 수 없다.

## 3. Spec 형식

### 계약 기반 그룹핑 (정의 → 계약 → 실현 → 보증)

Spec을 코드와의 계약서로 본다. 용어를 정의하고, 계약을 맺고, 실현 방법을 기술하고, 보증 조건을 명시한다.

```markdown
---
id: SPEC-001
title: 기능명
status: draft | approved | implementing | shipped | superseded
superseded_by: SPEC-XXX
related_specs: [SPEC-002]
author: 작성자
created: YYYY-MM-DD
updated: YYYY-MM-DD
---

# ── 정의 ──

# 1. 목적 (Purpose)
이 기능이 존재하는 이유. 한두 문장.

# 2. 용어 정의 (Glossary)
이 Spec 내에서 사용하는 도메인 용어의 정의.
| 용어 | 정의 |
|---|---|
| 예시 | ... |

# 3. 전제 (Assumptions)
이 Spec이 성립하기 위해 참이어야 하는 조건.
- 전제 1

# 4. 외부 의존성 (Dependencies)
| 대상 | 유형 | 버전/조건 | 장애 시 영향 |
|---|---|---|---|
| 예시 | 서드파티 | v1.0+ | ... |

# ── 계약 ──

# 5. 행위 명세 (Behavior)
기능의 동작 정의.
- 입력 → 출력
- 정상 흐름
- 경계 조건

# 6. 에러 처리 (Error Handling)
실패 시 동작 정의.
- 에러 유형별 복구/재시도/폴백 전략
- 사용자에게 보여줄 피드백
- 로깅/알림 기준

# 7. 범위 (Scope)
## 포함
- 이 Spec이 다루는 것
## 제외
- 이 Spec이 명시적으로 다루지 않는 것

# 8. 제약 (Constraints)
보안, 호환성, 규정 등 지켜야 할 조건.

# 9. 성능 기준 (Performance Criteria)
| 항목 | 기준 | 측정 방법 |
|---|---|---|
| 예시 | < 200ms | Instruments |

# ── 실현 ──

# 10. 기술 설계 (Technical Design)
## 10.1 구조
- 모듈 / 클래스 / 파일 구성
- 의존 관계
## 10.2 인터페이스
- 공개 API / 함수 시그니처 / 프로토콜
- 데이터 구조 (입출력 타입)
## 10.3 흐름
- 호출 순서 / 데이터 흐름
- 상태 전이 (해당 시)

# 11. 플랫폼별 차이 (Platform Notes)
## iOS
- 구현 차이, OS 버전 제약, 프레임워크 선택
## Android
- 구현 차이, minSdk, 라이브러리 선택

# 12. 마이그레이션 (Migration)
기존 코드/데이터에서 새 구현으로의 전환 방법.
- 전환 단계
- 하위 호환성
- 롤백 방법

# ── 보증 ──

# 13. 테스트 전략 (Test Strategy)
## 단위 테스트
- 대상, 커버리지 기준
## 통합 테스트
- 대상, 시나리오
## E2E / UI 테스트
- 대상, 시나리오

# 14. 수락 기준 (Acceptance Criteria)
- [ ] 기준 1
- [ ] 기준 2

# 15. 변경 이력 (Changelog)
| 날짜 | 변경 내용 | 사유 |
|---|---|---|
| YYYY-MM-DD | 초안 작성 | - |
```

### 그룹별 역할

| 그룹 | 역할 | SOT로서의 의미 |
|---|---|---|
| **정의** (1-4) | 이 Spec이 다루는 세계를 정의 | 전제가 깨지면 Spec 전체를 재검토해야 함을 알 수 있음 |
| **계약** (5-9) | 코드가 반드시 만족해야 할 조건 | 코드 리뷰 시 이 그룹만 대조하면 Spec 준수 여부를 판단 가능 |
| **실현** (10-12) | 계약을 이행하는 구체적 방법 | 코드의 구조가 이것과 다르면 Spec 개정 필요 |
| **보증** (13-15) | 계약 이행을 증명하는 방법 | 수락 기준 전항 통과가 shipped 전이의 게이트 |

### 깊이 가변 원칙

15개 섹션 모두 항상 존재한다. 작업 규모에 따라 각 섹션의 밀도가 달라진다.

- 가벼운 UI 변경: 각 섹션이 한 줄. 전체 Spec이 반 페이지.
- 결제/인증 등 민감 로직: 행위 명세에 경계 조건이 상세하고, 기술 설계에 시퀀스 다이어그램이 포함됨. 수 페이지.

구조는 같고, 밀도만 다르다.

## 4. Spec 작성 skill (`spec-writing`)

### 동작 방식: 대화형 탐색

- 개발자가 기능의 목적/맥락을 말하면, skill이 이를 파악한 뒤 필요한 질문을 판단해서 던진다.
- 섹션 순서에 얽매이지 않고, 대화 흐름에 따라 정보를 수집한다.
- 최종 산출물은 항상 위의 Spec 형식(15개 섹션, 계약 기반 4그룹)으로 수렴한다.
- 모든 섹션이 채워졌는지 검증 후 Spec 문서를 생성한다.

### Spec 작성 skill이 하는 것

1. 개발자의 초기 설명에서 목적과 맥락을 파악
2. 부족한 정보를 판단하여 질문 (한 번에 하나씩)
3. 수집된 정보를 Spec 형식의 각 섹션에 배치
4. 모든 섹션이 채워졌는지 검증
5. Spec 문서(draft 상태) 생성

### Spec 작성 skill이 하지 않는 것

- 섹션 순서를 강제하지 않음 (질문 순서는 맥락에 따라 유동적)
- 구현 방법을 결정하지 않음 (그것은 기술 설계 섹션에서 개발자가 작성)
- Spec을 승인하지 않음 (승인은 파이프라인의 다음 단계)

## 5. Spec 상태 머신

```
draft → approved → implementing → shipped
                                    ↓
                                superseded (by SPEC-XXX)
```

| 상태 | 의미 | 전이 조건 |
|---|---|---|
| `draft` | 작성 중 | spec-writing skill이 문서 생성 시 |
| `approved` | 리뷰 완료, 구현 가능 | 리뷰어 승인 |
| `implementing` | 구현 진행 중 | 구현 단계 진입 시 |
| `shipped` | 구현 완료, 코드와 일치 | 수락 기준 전항 통과 |
| `superseded` | 새 Spec으로 대체됨 | 후속 Spec이 approved될 때, superseded_by에 후속 Spec ID 기록 |

---

## 6. 플러그인 구성 요소

| 구성 요소 | 역할 |
|---|---|
| `feature` 커맨드 | Spec 중심 파이프라인의 진입점. 단계와 게이트를 강제 |
| Spec 형식 (템플릿) | 15개 섹션, 계약 기반 4그룹. SOT의 구조 |
| `spec-writing` skill | 대화형 탐색으로 Spec 작성을 안내 |
| Spec 상태 머신 | draft → approved → implementing → shipped / superseded |
| TDD 판단 게이트 | 구현 직전에 제안 + 사용자 판단, 결과를 Spec에 기록 |
| Spec ↔ 코드 일치 게이트 | 구현 중 Spec과 코드의 불일치를 방지 |

---

## 7. 범위

### 이 설계가 다루는 것

- `feature` 커맨드의 파이프라인 구조
- Spec 형식과 그룹핑
- Spec 작성 skill의 동작 방식
- Spec 상태 머신
- TDD 판단 게이트의 위치와 역할

### 이 설계가 다루지 않는 것

- `debug` 커맨드 (추후 별도 설계)
- Spec ↔ 코드 일치 게이트의 기술적 구현 방법 (구현 계획에서 다룸)
- 플러그인의 설치/배포 방법
- CI/CD 통합
