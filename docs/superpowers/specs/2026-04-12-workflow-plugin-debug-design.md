# workflow-plugin `debug` 커맨드 설계

> 상태: 승인 대기 (2026-04-12)

## 1. 정체성

### 목적

버그 수정을 가설 기반 파이프라인으로 구조적으로 조형한다.

### 핵심 원칙

- **구조적 조형.** `feature` skill과 동일한 철학. 디버깅 워크플로우의 형태를 강제한다. 단계를 안내하거나 추천하는 것이 아니라, 구조를 보증한다.
- **고정 구조, 가변 깊이.** 모든 디버깅이 동일한 8단계 파이프라인을 통과한다. 각 단계의 밀도만 버그 규모에 따라 달라진다. 압축은 가능, 스킵은 불가.
- **가설-검증 사이클.** 원인을 추측하고 바로 수정하는 것을 구조적으로 방지한다. 가설을 세우고 → 검증하고 → 확정된 원인만 수정한다.
- **회귀 테스트 강제.** 회귀 테스트는 판단 게이트(사용자 선택)가 아닌 필수 단계다. 디버깅의 본질은 재발 방지이므로, 구조에 박혀 있다.
- **Spec 비침범.** debug skill은 Spec을 직접 개정하지 않는다. Spec과의 불일치를 발견하고 기록(플래그)만 한다. 실제 Spec 개정은 `feature` 파이프라인의 정규 절차를 통해 별도로 진행한다.

### feature skill과의 관계

| 측면 | feature | debug |
|---|---|---|
| 파이프라인 | Spec 작성 → 승인 → 계획 → TDD 판단 → 구현 → 검증 → 완료 | 증상 기록 → Spec 확인 → 가설 수립 → 검증 → 원인 확정 → 수정 → 회귀 테스트 → 완료 |
| Spec 관계 | Spec을 **생성·개정**한다 | Spec을 **참조**만 한다. 불일치 발견 시 플래그 기록 |
| 판단 게이트 | TDD 적용 여부 (사용자 판단) | 없음. 회귀 테스트는 강제 |
| 산출물 | Spec + 코드 | 수정 코드 + 회귀 테스트 + 디버깅 기록 + Spec 불일치 플래그(해당 시) |

## 2. 가설 기반 파이프라인

### 단계 흐름

```
[1. 증상 기록] → [2. Spec 확인] → [3. 가설 수립] ⇄ [4. 검증] → [5. 원인 확정] → [6. 수정] → [7. 회귀 테스트] → [8. 완료]
```

3↔4단계 사이에 가설-검증 반복 루프가 존재한다. 가설이 기각되면 3단계로 돌아가 새 가설을 세운다. 이것은 허용된 루프이며, "스킵"이 아니라 "가설이 틀렸으므로 다시 세우는 것"이므로 고정 구조 원칙에 위배되지 않는다.

**이 파이프라인은 무조건 순서대로 진행한다. 어떤 단계도 건너뛸 수 없다.**

### 각 단계 정의

| 단계 | 행위 | 산출물 |
|---|---|---|
| **1. 증상 기록** | 버그 증상을 구체적으로 기록 | 증상 기록 (기대 동작 vs 실제 동작, 재현 조건) |
| **2. Spec 확인** | 해당 기능의 Spec 존재 여부 및 불일치 대조 | Spec 확인 결과 ("있음/없음", 불일치 여부) |
| **3. 가설 수립** | 원인 가설을 세우고 검증 방법을 함께 기록 | 가설 + 검증 계획 |
| **4. 검증** | 가설의 검증 방법을 실행 | 검증 결과 ("확인됨" 또는 "기각됨 — 이유") |
| **5. 원인 확정** | 검증된 가설을 원인으로 확정, Spec 불일치 판단 | 원인 기록 + Spec 불일치 플래그(해당 시) |
| **6. 수정** | 확정된 원인에 대한 수정 코드 작성 | 수정 코드 |
| **7. 회귀 테스트** | 재발 방지 테스트 작성 및 실행 (강제) | 회귀 테스트 코드 |
| **8. 완료** | 디버깅 기록 정리, Changelog 기록, 커밋 | 디버깅 기록 + 커밋 |

---

## 3. 각 단계 상세

### 1단계: 증상 기록

버그의 증상을 구체적으로 기록한다.

**기록 항목:**
- 기대 동작: 어떻게 동작해야 하는가
- 실제 동작: 실제로 어떻게 동작하는가
- 재현 조건: 환경, 입력, 순서, 빈도
- 영향 범위: 어떤 기능/화면에 영향을 주는가

**다음 단계로의 전이 조건:** 기대 동작, 실제 동작, 재현 조건이 기록되었다.

### 2단계: Spec 확인

해당 기능의 Spec이 존재하는지 확인하고, 존재하면 증상과 대조한다.

**행위:**
- `specs/` 디렉터리에서 관련 Spec을 탐색한다 (Glob 패턴: `specs/SPEC-*.md`).
- Spec이 있으면: 증상이 Spec의 행위 명세(5), 에러 처리(6)와 불일치하는지 대조한다.
  - "Spec 대로 동작하는데 기대와 다른 것" → Spec 자체의 결함 가능성
  - "Spec과 다르게 동작하는 것" → 코드의 결함
- Spec이 없으면: "Spec 없음"으로 기록하고 계속 진행한다. 디버깅은 Spec 없이도 가능하다.

**다음 단계로의 전이 조건:** Spec 확인 결과가 기록되었다 ("있음 — 불일치/일치" 또는 "없음").

### 3단계: 가설 수립

증상과 Spec 확인 결과를 바탕으로 원인 가설을 세운다.

**가설의 조건:**
- 구체적이어야 한다: "[파일:함수]에서 [조건]일 때 [잘못된 동작]이 발생할 것"
- 검증 방법이 함께 기록되어야 한다: "이 가설이 맞다면, [검증 방법]으로 확인할 수 있다"
- 추상적 가설을 금지한다: "아마 이 함수가 문제일 것" (X)

**좋은 가설 예시:**
- "`PaymentManager.applyCoupon()`에서 쿠폰 만료 여부를 체크하지 않아, 만료된 쿠폰이 할인에 적용된다. 검증: 만료된 쿠폰으로 `applyCoupon()`을 호출하면 할인이 적용되는지 확인."
- "`UserDefaults`에 다크모드 설정이 저장될 때 키 이름이 `isDarkMode`와 `darkMode`로 불일치한다. 검증: 두 키를 모두 읽어서 값이 다른지 확인."

**다음 단계로의 전이 조건:** 구체적 가설과 검증 방법이 기록되었다.

### 4단계: 검증

가설의 검증 방법을 실행한다.

**행위:**
- 가설에 기록된 검증 방법을 실행한다: 로그 확인, 브레이크포인트, 단위 테스트, 값 추적 등.
- 검증 결과를 기록한다:
  - "가설 확인됨 — 근거: [확인 결과]" → **5단계로 진행**
  - "가설 기각됨 — 이유: [기각 근거]" → **3단계로 돌아가 새 가설을 세운다**

**가설-검증 반복:**

```
[3. 가설 수립] → [4. 검증] → 확인됨 → [5. 원인 확정]
                     ↓
                  기각됨 → [3. 가설 수립] (새 가설)
```

이 반복은 파이프라인 안의 허용된 루프다. 기각된 가설은 기록으로 남겨 같은 가설을 반복하지 않도록 한다.

**다음 단계로의 전이 조건:** 가설이 확인되었다.

### 5단계: 원인 확정

검증된 가설을 원인으로 확정한다.

**기록 항목:**
- 어떤 코드가 (파일, 함수, 줄 번호)
- 왜 (로직 오류, 누락, 잘못된 조건 등)
- 어떤 조건에서 (입력, 상태, 타이밍)
- 버그를 발생시키는가

**Spec 불일치 판단:**
- Spec이 있으면: 원인이 Spec 불일치에 해당하는지 판단한다.
  - Spec과 코드가 불일치: 플래그를 기록한다.
    > "Spec 불일치 발견: SPEC-XXX [섹션명] — [구체적 불일치 내용]. feature 파이프라인에서 Spec 개정 필요."
  - Spec 자체에 결함이 있음 (Spec대로 동작하지만 그것이 잘못됨): 플래그를 기록한다.
    > "Spec 결함 발견: SPEC-XXX [섹션명] — [구체적 결함 내용]. feature 파이프라인에서 Spec 개정 필요."
- Spec이 없으면: 해당 없음.

**다음 단계로의 전이 조건:** 원인이 확정되고 기록되었다.

### 6단계: 수정

확정된 원인에 대한 수정 코드를 작성한다.

**행위:**
- 수정 범위는 확정된 원인에 한정한다. 주변 코드 정리나 리팩토링은 하지 않는다.
- 수정이 증상을 해결하는지 확인한다 (1단계의 재현 조건으로 검증).

**다음 단계로의 전이 조건:** 수정 코드가 작성되었고, 증상이 해소되었다.

### 7단계: 회귀 테스트 (강제)

이 버그가 재발하지 않음을 검증하는 테스트를 작성한다. **이 단계는 스킵할 수 없다.**

**행위:**
- 1단계의 재현 조건을 반영한 테스트를 작성한다.
- 테스트가 수정 전 코드에서 실패하고, 수정 후 코드에서 통과하는지 확인한다 (가능한 경우).
- 경계 조건이 원인이었다면 관련 경계 조건 테스트도 추가한다.

**다음 단계로의 전이 조건:** 회귀 테스트가 작성되고 통과했다.

### 8단계: 완료

디버깅 기록을 정리하고 마무리한다.

**행위:**
- 디버깅 기록을 정리한다:
  - 증상, 가설들(기각된 것 포함), 검증 결과, 원인, 수정 내용, 회귀 테스트
- 해당 Spec이 있으면 Spec의 Changelog에 버그 수정 기록을 남긴다:
  - "[날짜] 버그 수정: [증상 요약] — 원인: [원인 요약]"
- 커밋한다.
- 개발자에게 완료를 알린다. Spec 불일치 플래그가 있으면 포함:
  > "버그 수정이 완료되었습니다. SPEC-XXX의 [섹션]에 불일치가 발견되었으므로, feature 파이프라인에서 Spec 개정이 필요합니다."
- Spec 불일치 플래그가 없으면:
  > "버그 수정이 완료되었습니다."

## 4. 진입

개발자가 debug를 시작할 때의 진입 로직.

### Spec 유무에 따른 진입

- **Spec이 있는 기능의 버그:** 모든 단계가 Spec을 참조하며 진행. 2단계에서 Spec 대조, 5단계에서 불일치 플래그.
- **Spec이 없는 기능의 버그 (레거시 등):** 2단계에서 "Spec 없음"으로 기록하고 계속 진행. 5단계에서 Spec 불일치 판단 건너뜀. 나머지 단계는 동일.

### 대화 단절 시 상태 복원

`feature`와 달리 debug는 Spec의 status를 변경하지 않으므로, 상태 복원은 디버깅 기록 파일에 의존한다. 새 대화가 시작되면:
- 진행 중인 디버깅 기록이 있는지 확인한다.
- 기록의 마지막 완료 단계를 파악하여 해당 단계 다음부터 재개한다.

## 5. 역방향 전이

### 허용되는 역방향 전이

| 현재 단계 | 되돌아갈 단계 | 조건 |
|---|---|---|
| 4(검증) → | 3(가설 수립) | 가설 기각됨 (허용된 루프) |
| 6(수정) → | 3(가설 수립) | 수정 후에도 증상이 해소되지 않음 — 원인이 잘못 확정되었을 가능성 |
| 7(회귀 테스트) → | 6(수정) | 회귀 테스트 작성 중 수정의 부작용 발견 |

### 역방향 전이의 원칙

- 항상 기록한다: 왜 되돌아가는지, 어떤 판단에 의한 것인지.
- 기각된 가설은 기록으로 남겨 같은 가설을 반복하지 않는다.
- 건너뛰기는 허용하지 않는다. 6→3으로 갈 때 4, 5를 다시 밟는다.

## 6. 깊이 가변 원칙의 적용

이 파이프라인의 모든 단계는 버그 규모와 무관하게 항상 실행된다. 달라지는 것은 각 단계의 **밀도**다.

**가벼운 버그 (예: 오타, 단순 로직 오류):**
- 증상 기록: "버튼 텍스트가 '확인'이어야 하는데 '확이'으로 표시됨"
- Spec 확인: "해당 Spec 없음" 또는 "Spec과 무관"
- 가설: "`MainView.swift:42`의 문자열 오타" → 검증: 확인 → 1회 사이클
- 수정: 한 줄
- 회귀 테스트: 문자열 검증 테스트 1개
- 전체 과정: 수 분

**무거운 버그 (예: 간헐적 크래시, 레이스 컨디션):**
- 증상 기록: 재현 조건 상세 (환경, 빈도, 스택 트레이스, 디바이스별 차이)
- Spec 확인: 행위 명세(5)와 대조, 경계 조건 확인, 에러 처리(6) 대조
- 가설: 여러 가설 → 기각 → 새 가설... 3-4회 반복
- 수정: 다수 파일
- 회귀 테스트: 통합 테스트 + 경계 조건 테스트 + 동시성 테스트
- 전체 과정: 수 시간~수 일

**구조는 같고, 밀도만 다르다.**

## 7. 플러그인 구성 요소 (추가)

기존 플러그인에 아래 구성 요소가 추가된다:

| 구성 요소 | 역할 |
|---|---|
| `debug` skill | 가설 기반 디버깅 파이프라인의 진입점. 8단계와 가설-검증 루프를 강제 |

### 기존 구성 요소와의 관계

- **`feature` skill:** debug는 feature의 Spec을 참조하지만 개정하지 않는다. 불일치 플래그만 남기고, 개정은 feature 파이프라인에서 처리.
- **Spec 형식 (템플릿):** debug가 직접 Spec을 생성하지 않으므로 템플릿을 사용하지 않는다.
- **`spec-writing` skill:** debug에서 호출하지 않는다.

---

## 8. 범위

### 이 설계가 다루는 것

- `debug` 커맨드의 파이프라인 구조
- 가설-검증 사이클의 동작 방식
- 회귀 테스트 강제의 위치와 역할
- Spec 비침범 원칙과 불일치 플래그
- `feature` skill과의 관계

### 이 설계가 다루지 않는 것
