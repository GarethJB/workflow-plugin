스펙 주도 개발 (SDD) 란?

Spec-Driven Development

코드를 먼저 쓰고 문서를 뒤에 맞추는 대신,
Spec을 합의의 중심 축으로 삼고 구현·테스트·리뷰가 그 위에서 돌아가게 하는 방법론.

전통 방식의 한계

  • 코드가 유일한 SOT가 되면서 "왜"가 소실
  • 요구사항이 기획서·회의·팀채팅 등에 흩어짐
  • AI/신규 팀원이 맥락을 재구성하기 어려움

SDD가 바꾸는 것

  • Spec이 단일 진실 원천(SOT)
  • 구현 전 수락 기준을 합의
  • AI가 Spec을 읽고·따르고·갱신

SDD의 핵심 원칙 8가지 (1/2)

Shen Li의 원문을 기반으로 프로젝트 관점에서 재서술

1. 맥락이 생산성이다

기존 API·데이터 구조·설계 패턴을 문서로 제공해야 AI가 프로젝트 관행을 이해한다.

2. 명령은 구조화한다

문제·사용자 흐름·기술 요구·수락 기준을 명시한다. 빈칸이 많은 요청은 빈칸이 많은 결과를 낳는다.

3. 살아있는 프로젝트 핸드북

프로젝트 관행을 한 문서에 누적해 신규 팀원(사람이든 AI든)의 진입점으로 삼는다.

4. AI는 유능하지만 맥락이 부족한 팀원

지시는 구체적으로, 결과는 반드시 검증한다.

참고 — Spec-Driven Development (SDD) is the Future of Software Engineering · Shen Li, Medium

SDD의 핵심 원칙 8가지 (2/2)

5. 코드보다 문서가 먼저

설계·요구·아키텍처를 구현 이전에 써둔다. 문서가 합의의 결과이자 구현의 씨앗.

6. SDD는 천장이 아니라, 바닥을 올린다

개발자가 문제를 이해한 상태에서만 AI가 생산성을 끌어올린다. 아키텍처·UX의 주도권은 사람.

7. Accept-All 금지

읽지 않고 승인하면 품질이 무너진다. 결정마다 사람의 이해와 검증이 동반된다.

8. 패턴을 맥락으로 환류

기능 완료 후 발견된 패턴은 프로젝트 핸드북에 반영해 다음 기능이 그 위에서 시작하게 한다.

핵심 원칙 9

  1. Spec이 SOT다. 엇갈리면 Spec이 기준.
  2. 15섹션 고정 구조. 4그룹(정의 / 계약 / 실현 / 보증).
  3. Spec은 spec-writing 스킬로만 편집. 훅+락으로 직접 편집 차단.
  4. 수락 기준은 번호로. AC-NN·§N을 커밋·이슈·테스트에 그대로.
  5. AI는 대신 결정하지 않는다. 복수 안 제시, 개발자가 선택.
  6. 단계마다 Changelog에 마커. 재개 지점 복원.
  7. 계획이 먼저, 구현은 나중.
  8. 프로젝트 전역 맥락은 CLAUDE.md. 자동 로드되는 살아있는 핸드북.
  9. 요청은 5필드로 구조화. 문제·맥락·기대 결과·제약·산출물.

프로젝트 전역 맥락 · CLAUDE.md

Claude Code가 모든 세션에 자동 주입하는 살아있는 핸드북.

CLAUDE.md에 담는 것

  • 아키텍처 레이어·모듈 경계
  • API 카탈로그·외부 의존성
  • 상태 관리·에러 전파 규칙
  • 프로젝트 전역 테스트 계층
  • 스타일·네이밍 컨벤션

기능 Spec과의 관계

  • 일반 Spec은 CLAUDE.md의 규약을 반복 기술하지 않는다
  • §4·§10.1은 해당 Spec 고유의 결정만 기록
  • feature 7단계: 새 패턴 발견 시 pattern-extracted: 마커만 기록
  • CLAUDE.md도 SOT — 파이프라인은 직접 수정하지 않고 사람이 별도 리뷰로 반영 결정

구조화된 요청 5필드

모호한 요청은 스킬도 구제 못 한다

#필드내용
1문제무엇이 필요한가 / 무엇이 고장났는가. 한 문장.
2맥락관련 Spec·AC-NN, CLAUDE.md의 전역 규약
3기대 결과"완료"의 관찰 가능한 정의
4제약시간·플랫폼·위험도·금지 사항
5산출물 형식Spec / 구현 커밋 / 가설 목록 / 분석 보고

상세 템플릿과 예시: docs/structured-request.md

feature 파이프라인 (7단계)

  1. Spec 작성 — spec-writing 스킬 호출
  2. 승인 — Spec 리뷰 및 확정 (spec_flags 소비)
  3. 계획 — 구현 단계 세분화, §10 완성
  4. Practice 선택 — TDD (고) / Walking Skeleton (중) / Implementation First (저)
  5. 구현 — 선택한 Practice의 규칙에 따라
  6. 검증 — AC-NN 단위 통과 여부 확인
  7. 완료 — Changelog 기록, 락 해제

debug 파이프라인 (8단계)

  1. 증상 기록 — 재현 조건·기대값·실제값
  2. Spec 확인 — 어느 §·AC가 깨졌는가
  3. 가설 수립 — 원인 후보 나열
  4. 검증 — 가설별 실험 설계
  5. 원인 확정 — spec_flags 기록
  6. 수정 — 코드 혹은 Spec 교정
  7. 회귀 테스트 — §13 전략에 따라 추가
  8. 완료 — Changelog 기록

Practice 선택 기준

위험도 3단계와 1:1로 대응

위험도Practice적합한 작업
고TDD결제·인증·데이터 정합성 등 회귀 민감, 로직 복잡
중Walking Skeleton새 SDK·프로토콜·통신 경로 개척 (통합 리스크)
저Implementation First내부 도구·프로토타입, 검증 범위 작음

혼용 가능 — 예: 관통 경로는 Walking Skeleton, 내부 도메인 로직은 TDD. feature 4단계에서 확정·Changelog 기록.

운영 규약

락 파일

.spec-writing-active 등이 남아 있으면 다른 세션이 작업 중일 수 있음. 고아 락이 확실할 때만 제거.

AC-NN · §N 표기

커밋 메시지·이슈·테스트명에서 동일 표기 사용. "어디가 왜 실패했는가"가 검색 가능해진다.

[AI 제안] 태그

§10.2·10.3에서 AI가 생성한 안은 [AI 제안] 태그를 달고, 개발자가 선택·수정 후 태그를 제거한다.

Changelog 마커

각 파이프라인 단계 종료 시 §15에 마커 한 줄. 세션이 중단돼도 다음 세션이 재개 지점을 찾을 수 있다.

진입 후에는 스킬이 단계별로 질문을 던진다. 그 흐름을 따라가면 된다.

기대 효과

  • 리뷰 비용 감소 — 15섹션 고정으로 "어디를 봐야 하는가"가 일정함
  • AI 드리프트 차단 — Spec이 SOT, 코드가 Spec과 어긋나면 즉시 감지
  • 버그 추적성 — AC-NN·§N으로 증상·원인·수정의 경로가 선형화
  • 재개 가능성 — Changelog 마커로 세션 단절에 강함
