# 구조화된 요청 체계 (Structured Request)

이 플러그인의 스킬을 트리거할 때, **개발자가 AI에게 던지는 초기 요청 자체가 구조화되어 있어야** 파이프라인이 제 성능을 낸다. "뭔가 로그인 기능 만들어줘"식의 모호한 요청은 스킬이 커버할 수 있는 범위를 넘어서는 근본 원인이다 — 스킬은 *구조를 강제*할 수 있어도 *맥락 부재*는 보완하지 못한다.

## 요청 템플릿 (5 필드)

모든 요청은 다음 5개 필드를 명시적으로 포함한다. 정보가 없으면 "없음" 또는 "미정"으로라도 기재한다 — 공백 자체가 탐색 지점이 된다.

```
1. 문제 (Problem)
   지금 무엇이 필요한가 / 무엇이 고장났는가. 한 문장.

2. 맥락 (Context)
   - 관련 Spec: SPEC-XXX §N (또는 "신규 Spec 필요")
   - 관련 AC: AC-NN (이미 있으면)
   - 프로젝트 제약: CLAUDE.md의 해당 항목 (아키텍처·의존성·전역 규약)

3. 기대 결과 (Desired Outcome)
   "완료"의 관찰 가능한 정의. 수락 기준 후보.

4. 제약 (Constraints)
   시간·플랫폼·위험도·금지 사항. 없으면 "없음".

5. 산출물 형식 (Output)
   Spec 작성 / 구현 PR / 가설 목록 / 분석 보고 중 어느 것인가.
```

## 경로별 기본 매핑

| 상황 | 권장 스킬 | 최소 필요 필드 |
|---|---|---|
| 새 기능 착수 | `feature` | 1, 2(기획 문서 또는 CLAUDE.md 참조), 3, 5="신규 Spec + 구현 PR" |
| 버그 수정 | `debug` | 1, 2(재현 조건·관련 AC-NN), 3(기대 동작), 5="원인 보고 + 패치" |
| Spec 신규 작성 | `spec-writing` 경로 A/B | 1, 2(기획서 경로) OR 대화형(경로 B), 3, 5="spec.md" |
| Spec 수정 | `spec-writing` 경로 C | 1, 2(대상 Spec·섹션), 3, 5="diff" |
| 탐색·질문 | 스킬 없이 대화 | 1, 2, 5="답변" |

## 좋은 예 / 나쁜 예

**나쁜 예:**
> "로그인 좀 고쳐줘"

- 문제 불명 (무엇이 고장났는가?)
- 맥락 없음 (어느 Spec·AC?)
- 기대 결과 없음
- 출력 형식 불명

**좋은 예 (debug 스킬 진입):**
> "SPEC-003 §5 AC-05 실패. 카카오 로그인 후 토큰 갱신 시점에 홈 화면이 빈 상태로 뜬다(재현 100%). CLAUDE.md의 비동기 초기화 순서 규약 적용 가능성 확인 필요. 원인 가설 3개와 각 검증 방법을 제시해줘. 패치는 승인 후."

- 필드 1: 홈이 비는 문제
- 필드 2: SPEC-003 AC-05, CLAUDE.md 비동기 초기화 규약
- 필드 3: 가설 3개 + 검증 방법
- 필드 4: 패치는 승인 후 (제약)
- 필드 5: 원인 보고

**좋은 예 (feature 스킬 진입):**
> "'앱 내 공지 배너' 기능을 추가. 관리자가 서버에 등록한 공지를 홈 최상단에 노출, 사용자가 닫으면 7일간 재노출 안 함. CLAUDE.md의 홈 화면 섹션 레이아웃·SharedPreferences 키 네이밍 규약을 따른다. 중위험(새 엔드포인트 1개 + 상태 지속). 신규 Spec 작성 후 승인받고 구현 PR까지 — 7일 재노출 로직의 경계 조건을 §5 경계 조건에 상세히 기술해줘."

- 필드 1: 홈 공지 배너 추가
- 필드 2: CLAUDE.md 홈 레이아웃·Prefs 네이밍, (신규 Spec 예정이라 AC는 아직 없음), 중위험
- 필드 3: 관리자 공지가 홈 상단에 노출 / 닫기 후 7일 재노출 없음
- 필드 4: 없음
- 필드 5: 신규 Spec + 구현 PR

## 안티패턴

플러그인 맥락에서 특히 자주 보이는, 언뜻 구조화된 듯 보이지만 실제론 맥락이 빠져 있는 패턴들.

1. **스킬 이름만 던지기.** "feature 스킬로 시작해줘"만 보내고 5필드 없이 진행 — 스킬이 5필드를 역으로 캡처하겠지만 사이클이 2~3배로 길어진다. 최소 필드 1·3·5는 먼저 명시.
2. **이전 세션 기억에 의존.** "저번에 말한 그 로그인 개선". 세션이 바뀌면 맥락은 휘발된다. 관련 Spec ID·AC-NN을 매번 필드 2에 다시 적는다.
3. **파이프라인 단계를 한 번에 요청.** "Spec 쓰고 바로 구현까지 해줘". 이 플러그인은 *스킵 불가* 파이프라인이 본질. 단계를 묶어 요청해도 스킬은 분리 진행하므로 혼란만 늘어난다.
4. **Spec 없이 바로 구현 요청.** "로그인 버튼 색만 바꿔줘". 한 줄 수정이라도 `feature` 진입이 원칙 — 색상 결정이 CLAUDE.md 스타일 규약과 충돌하는지는 스킬을 거쳐야 걸러진다. 정말 트리비얼하면 스킬 없이 대화로 진행하되, Spec에 영향을 주는 변경은 아무리 작아도 파이프라인으로.
5. **Accept-All 태도.** AI가 생성한 `[AI 제안]` 태그를 검토 없이 제거·승인 — 원문 SDD 아티클이 가장 경계하는 실수. 태그를 지우기 전 **본인 언어로 왜 그 안을 골랐는지** 한 줄이라도 Changelog나 PR 본문에 남기는 습관.

## 언제 느슨해도 되는가

- **탐색 단계의 자유 질문**("이 프로젝트 설명해줘")은 구조화 대상이 아니다. 이건 온보딩 영역.
- **가벼운 작업**(프로토타입 수준)은 필드 2·4를 "없음"으로 둘 수 있다. 하지만 1·3·5는 여전히 명시한다.

## 플러그인과의 연결

- `feature`·`debug` 스킬은 진입 시 위 5필드를 역으로 캡처한다 — 사용자가 구조화된 요청을 하지 않았어도 스킬이 질문으로 메운다. 다만 **처음부터 구조화해서 던질수록 사이클이 짧아진다.**
- CLAUDE.md의 관련 항목을 필드 2에 관례적으로 포함하면 프로젝트 전역 제약(아키텍처·API 패턴)이 누락되는 사고를 줄인다. Claude Code가 세션마다 자동 로드하지만, 요청에 명시하면 LLM이 더 확실히 참조한다.

### 요청 ↔ Spec 15섹션 매핑

요청이 곧 Spec의 씨앗이다. 아래 매핑이 성립하므로, **요청서를 그대로 Spec 초안의 1차 재료**로 재사용할 수 있다. 이는 `spec-writing` 경로 B(대화형 신규)의 첫 번째 탐색 단계를 단축한다.

| 요청 필드 | Spec 섹션 | 비고 |
|---|---|---|
| 1 · 문제 | §1 목적 (Job Story) | "[사용자]가 [목적]을 달성하기 위해 [기능]을 한다" 형식으로 승격 |
| 3 · 기대 결과 | §14 수락 기준 (AC-NN) | 관찰 가능한 정의 → AC-NN 후보. 1:N 매핑 가능 |
| 2 · 관련 Spec·AC | frontmatter `related_specs`, §4 외부 의존성 일부 | 타 Spec·외부 계약 참조 |
| 2 · CLAUDE.md 제약 | §10.1 구조(암묵 전제) · §8 제약 | 전역 규약을 어느 섹션에서 재진술할지는 해당 Spec 고유성만 |
| 4 · 제약 | §8 제약 · §9 성능 기준 | 시간·플랫폼·금지 사항은 §8, 성능 수치는 §9 |
| 4 · 위험도 힌트 | §15 Changelog의 Practice 선택 사유 | `feature` 4단계 Practice 판단 입력 |
| 5 · 산출물 형식 | (Spec 외) 파이프라인 메타 | 어떤 산출물이 나올지 — Spec 본문에는 들어가지 않음 |
