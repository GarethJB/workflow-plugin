# workflow-plugin 발표자료

## 네이티브 개발부서의 워크플로우를 구조적으로 조형하는 Claude Code 플러그인

---

## 1. 왜 만들었는가

### 문제

- 기능 개발의 흐름이 사람마다 다르다
- "이 기능 왜 이렇게 만들었지?"에 대한 답이 슬랙 어딘가에 묻혀 있다
- 기획서는 있지만, 기획서와 코드 사이의 간극을 메우는 문서가 없다
- 버그를 고칠 때 "아마 이것 때문일 거야" → 바로 수정 → 다른 문제 발생 패턴이 반복된다
- 온보딩 시 "어떤 순서로 일하면 되나요?"에 대한 구조화된 답이 없다

### 해결 방향

**워크플로우를 안내하거나 추천하는 게 아니라, 구조 자체를 강제한다.**

플러그인이 단계와 게이트를 정의하고, 모든 작업이 동일한 파이프라인을 통과하도록 만든다. 압축은 가능하지만 스킵은 불가능하다.

---

## 2. 핵심 원칙

### Spec = SOT (Source of Truth)

Spec이 소스코드의 단일 진실 공급원이다.

```
기획서 → [변환] → Spec (SOT) → [구현] → 코드
                    ↑                      |
                    └── 코드와 다르면 ──────┘
                        둘 중 하나를 맞춘다
```

- 코드는 Spec을 따른다
- 코드가 Spec과 다르면 코드가 틀리거나, Spec을 먼저 개정해야 한다
- "왜 이렇게 구현했지?" → Spec을 보면 된다

### 구조적 조형

| 기존 | workflow-plugin |
|---|---|
| "이렇게 하면 좋겠다" (안내) | "이 순서를 따른다" (강제) |
| 개발자가 프로세스 선택 | 플러그인이 프로세스 형태를 보증 |
| 스킵 가능 | **스킵 불가, 압축만 가능** |

### 고정 구조, 가변 깊이

모든 작업이 동일한 파이프라인을 통과한다. 달라지는 건 **밀도**뿐.

| | 가벼운 작업 (UI 수정) | 무거운 작업 (결제 로직) |
|---|---|---|
| Spec | 반 페이지 | 3~5 페이지 |
| 계획 | 작업 목록 | 모듈 단위 분해 |
| TDD | "아니오" | "예" |
| 검증 | 수락 기준 1~2개 | 다수 + 성능 기준 |
| **구조** | **동일** | **동일** |

## 3. 플러그인 구성

```
workflow-plugin/
├── skills/
│   ├── feature/SKILL.md        ← feature 파이프라인
│   ├── debug/SKILL.md          ← debug 파이프라인
│   └── spec-writing/
│       ├── SKILL.md            ← Spec 작성 skill
│       └── references/
│           └── analysis-strategy.md  ← 기획서 분석 전략
├── templates/
│   └── spec-template.md        ← Spec 형식 (15섹션, 4그룹)
├── specs/                      ← Spec 저장소
└── debug-logs/                 ← 디버깅 기록 저장소
```

## 4. Spec 형식

### 계약 기반 4그룹, 15개 섹션

```
┌─────────────────────────────────────────────────┐
│  정의 (Definition)                               │
│  1. 목적  2. 용어 정의  3. 전제  4. 외부 의존성   │
├─────────────────────────────────────────────────┤
│  계약 (Contract)                     ← 핵심     │
│  5. 행위 명세  6. 에러 처리  7. 범위             │
│  8. 제약  9. 성능 기준                           │
├─────────────────────────────────────────────────┤
│  실현 (Realization)                              │
│  10. 기술 설계  11. 플랫폼별 차이  12. 마이그레이션│
├─────────────────────────────────────────────────┤
│  보증 (Assurance)                                │
│  13. 테스트 전략  14. 수락 기준  15. 변경 이력     │
└─────────────────────────────────────────────────┘
```

### 그룹별 역할

| 그룹 | 질문 | 코드 리뷰에서의 역할 |
|---|---|---|
| **정의** | 이 Spec이 다루는 세계는? | 전제가 깨졌으면 Spec 전체 재검토 |
| **계약** | 코드가 만족해야 할 조건은? | **이 그룹만 대조하면 Spec 준수 여부 판단 가능** |
| **실현** | 어떻게 구현하는가? | 구조가 다르면 Spec 개정 필요 |
| **보증** | 어떻게 증명하는가? | 수락 기준 전항 통과 = shipped |

### Spec 상태 머신

```
draft → approved → implementing → shipped
                                     ↓
                                 superseded
```

## 5. feature 파이프라인

### 7단계 고정 파이프라인

```
[Spec 작성] → [Spec 승인] → [계획] → [TDD 판단] → [구현] → [검증] → [완료]
    draft       approved     approved   approved   implementing       shipped
```

### 주요 게이트

**TDD 판단 게이트 (4단계)**
- 플러그인이 "TDD를 적용하시겠습니까?" 제안
- 개발자가 판단, 결과를 Changelog에 기록
- 자동 분류가 아닌 "제안 + 판단" — 판단해야 할 순간을 구조적으로 보장

**Spec ↔ 코드 일치 게이트 (5단계)**
- 구현 중 Spec과 괴리 발견 시: 코드를 맞추거나 Spec을 먼저 개정
- Spec과 코드가 불일치한 상태로 검증 단계 진입 불가
- 체크 타이밍: 커밋 직전, 새 모듈 생성 시, 예상과 다른 동작 발견 시

### 기획서 → Spec 변환

```
[기획서 첨부?]
    ↓ 예                        ↓ 아니오
[기획서 분석]                 [대화형 탐색]
    ↓                             ↓
[15개 섹션에 매핑]                 ↓
    ↓                             ↓
[빈 섹션 식별]                     ↓
    ↓                             ↓
[빈 섹션 보완 탐색] ←──────────────┘
    ↓
[Spec 생성 (draft)]
```

- 기획서에는 보통 **What은 있고 How는 없다**
- 기획서가 채우는 영역: 정의 + 계약의 일부 (정상 흐름)
- 보완이 필요한 영역: 경계 조건, 에러 처리, 기술 설계, 테스트 전략, 성능 기준
- PDF, Figma, 이미지, 마크다운 등 자유 형식 지원

## 6. debug 파이프라인

### 8단계 가설 기반 파이프라인

```
[증상 기록] → [Spec 확인] → [가설 수립] ⇄ [검증] → [원인 확정] → [수정] → [회귀 테스트] → [완료]
```

### 왜 가설 기반인가

```
❌ 기존 패턴:
   "아마 이것 때문일 거야" → 바로 수정 → 안 됨 → "그럼 저것?" → 수정 → 다른 문제 발생

✅ 가설 기반:
   가설 수립 (구체적) → 검증 (맞는지 확인) → 확정된 원인만 수정
   가설 기각 시 → 기록 남기고 → 새 가설 (같은 가설 반복 방지)
```

### feature와의 관계

| | feature | debug |
|---|---|---|
| Spec 관계 | Spec을 **생성/개정** | Spec을 **참조만** |
| Spec 변경 | 직접 변경 + 승인 | **불일치 플래그만 기록** |
| 판단 게이트 | TDD (사용자 선택) | 없음 (회귀 테스트 강제) |
| 산출물 | Spec + 코드 | 수정 코드 + 회귀 테스트 + 디버깅 기록 |

### Spec 비침범 원칙

debug는 Spec을 직접 개정하지 않는다.

```
[debug] → Spec 불일치 발견 → 플래그 기록 (debug-logs/)
                                    ↓
[feature] → 플래그 확인 → Spec 개정 → 승인 (정규 절차)
```

이유: debug가 Spec을 직접 고치면 feature의 승인 절차를 우회하게 되고, Spec의 SOT 권위가 훼손된다.

### 회귀 테스트 강제

- 판단 게이트가 아닌 **필수 단계**
- 오타 한 글자 수정이라도 회귀 테스트 작성 (깊이만 가변)
- 디버깅의 본질은 재발 방지

## 7. 적용 방법론

세 가지 방법론이 **구조에 녹아 있는 재료**로 작동한다. 사용자가 선택하는 옵션이 아니다.

| 방법론 | 적용 위치 | 방식 |
|---|---|---|
| **SDD** (Spec-Driven Development) | feature 파이프라인 전체 | 기본 기조. 모든 작업이 Spec으로 시작 |
| **Trunk-Based + PR** | 가벼운 작업 | 별도 경로가 아닌, 동일 구조의 **압축** |
| **TDD** | feature 4단계 (판단 게이트) | 제안 + 사용자 판단. 결제/인증 등에 권장 |

---

## 8. 대화 단절 시 상태 복원

긴 파이프라인은 한 대화에서 끝나지 않을 수 있다. 상태 복원 전략:

### feature

| Spec status | Changelog 기록 | 재개 지점 |
|---|---|---|
| `draft` | - | 2단계 (승인) |
| `approved` | 계획 확정 없음 | 3단계 (계획) |
| `approved` | 계획 확정 있음, TDD 판단 없음 | 4단계 (TDD 판단) |
| `implementing` | - | 5단계 (구현) |

### debug

- `debug-logs/DEBUG-XXX.md`의 `status: in_progress`
- 마지막 채워진 섹션으로 재개 지점 판단

## 9. 작업 예시 (요약)

### feature: "결제에 쿠폰 적용 기능 추가"

```
1. 기획서 첨부 → spec-writing이 분석 → 빈 섹션(에러 처리, 기술 설계 등) 보완 질문
2. SPEC-001 draft 생성 → 리뷰 → approved
3. 구현 계획 수립 → 확인
4. TDD 판단: "예" (결제 로직)
5. 테스트 우선 구현 → Spec 일치 확인하며 진행
6. 수락 기준 전항 통과
7. SPEC-001 shipped
```

### debug: "퍼센트 쿠폰 할인이 안 됨"

```
1. 증상: 퍼센트 쿠폰 적용 시 금액 변동 없음
2. SPEC-001 행위 명세와 대조 → 코드의 결함
3. 코드 탐색 → git blame → 3일 전 리팩토링에서 amount 누락 발견
   가설: "CouponCalculator에서 amount * (value/100) 대신 value/100만 반환"
4. 검증: 확인됨
5. 원인 확정: CouponCalculator.swift:28
6. 수정: 한 줄
7. 회귀 테스트 2개 작성 (퍼센트 할인 + 경계 조건)
8. 완료 (DEBUG-001 resolved)
```

## 10. 기대 효과

| 영역 | Before | After |
|---|---|---|
| **워크플로우 일관성** | 사람마다 다른 흐름 | 동일 파이프라인, 밀도만 다름 |
| **의사결정 추적** | 슬랙/구두에 흩어짐 | Spec + Changelog에 기록 |
| **기획-코드 간극** | 기획서와 코드 사이 공백 | Spec이 중간 다리 (SOT) |
| **디버깅 품질** | 짐작 수정 반복 | 가설-검증 사이클 강제 |
| **회귀 방지** | 테스트 선택적 | 회귀 테스트 강제 |
| **온보딩** | "어떻게 일해요?" | 플러그인이 구조를 안내 |
| **코드 리뷰** | "이거 왜 이래요?" | Spec 계약 그룹과 대조 |
