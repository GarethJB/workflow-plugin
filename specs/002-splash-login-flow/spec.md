# ── 정의 ──

# 1. 목적 (Purpose)

**재방문 사용자가 빠른 진입을 달성하기 위해 자동 로그인과 서버 자동 선택을 제공하고, 신규·서버 변경 사용자가 최초 접속을 완료하기 위해 서버 선택 → 소셜 로그인 → 가입 여부 분기 경로를 제공한다.**

앱 실행 시점부터 메인 화면(또는 온보딩 화면) 진입 직전까지의 진입 플로우를 정의한다.

> **성공 지표**: 미정 — 보완 조건: 운영팀이 전환·유지 KPI를 정의하는 시점.

# 2. 용어 정의 (Glossary)

| 용어 | 정의 |
|---|---|
| API URL | 사용자가 선택한 서버의 기본 URL. 모든 API 호출의 베이스. |
| 서버 목록 | `/api/v1/staff/list` 응답으로 조회되는 연결 가능한 서버 목록. 각 항목은 `serverName`, `apiUrl`, `imgServerUrl`, `photoUrl`, `photoUploadUrl`을 포함. |
| socialSlct | 소셜 로그인 공급자 식별자 (naver / kakao / google / apple). |
| socialKey | 소셜 SDK 인증 성공 시 발급받는 공급자별 식별 토큰. |
| JSESSIONID | 서버가 발급하는 세션 식별 쿠키. |
| remember-me | 자동 로그인용 영속 쿠키. |
| 가입 여부 code | `/auth/isRegistered` 응답 필드. `"-1"`(기존 회원) / `"1"`(미등록). |
| 자동 로그인 | 앱 재실행 시 저장된 쿠키의 존재만 확인하여(유효성 검증 없이) 메인 화면으로 이동하는 동작. |

# 3. 전제 (Assumptions)

- 각 소셜 SDK(네이버/카카오/구글/애플)가 앱에 사전 연동되어 있고 초기화되어 있다.
- 서버별 `/api/v1/staff/list`, `/api/v1/auth/isRegistered`, `/api/v1/auth/login` 엔드포인트가 동일한 계약으로 존재한다.
- 앱은 단일 로그인 세션만 유지한다. 두 계정 동시 로그인은 지원하지 않는다.
- iOS Keychain 및 Android EncryptedSharedPreferences를 사용할 수 있다.

# 4. 외부 의존성 (Dependencies)

| 대상 | 유형 | 버전/조건 | 장애 시 영향 |
|---|---|---|---|
| 네이버 로그인 SDK | SDK | 플랫폼별 공식 SDK | 네이버 로그인 경로 전체 실패 |
| 카카오 SDK | SDK | 플랫폼별 공식 SDK | 카카오 로그인 경로 전체 실패 |
| Google Sign-In SDK | SDK | 플랫폼별 공식 SDK | 구글 로그인 경로 전체 실패 |
| Apple Sign In API | SDK | iOS 13+ | 애플 로그인 경로 전체 실패 (iOS 한정) |
| `GET /api/v1/staff/list` | REST API | 서버별 배포 | 스플래시에서 서버 선택 불가 |
| `POST /api/v1/auth/isRegistered` | REST API | 서버별 배포 | 가입 여부 판별 불가 |
| `POST /api/v1/auth/login` | REST API | 서버별 배포 | 기존 회원 로그인 불가 |
| 로컬 암호화 저장소 | 로컬 저장소 | iOS Keychain / Android EncryptedSharedPreferences | 자동 로그인 동작 불가 |

# 5. 행위 명세 (Behavior)

## 정상 흐름

### 5.1 스플래시 진입

1. 앱 실행 → 스플래시 화면 표시.
2. 저장된 API URL 조회 및 `GET /api/v1/staff/list` 호출을 수행한다.
3. 저장된 API URL이 존재하면 해당 서버를 자동 선택(쿠키 초기화 없음).
4. 저장된 API URL이 없으면 서버 목록 UI를 표시하고 사용자 수동 선택을 대기한다.
5. 사용자가 서버를 수동 선택하면, 저장된 쿠키를 삭제하고 해당 서버의 API URL을 저장한다.

### 5.2 자동 로그인 판별

1. API URL 설정 직후, 로컬 저장소의 세션 쿠키(JSESSIONID/remember-me) 존재를 확인한다.
2. 쿠키가 존재하면 유효성 검증 없이 즉시 메인 화면으로 이동한다.
3. 쿠키가 없으면 로그인 화면으로 이동한다.

### 5.3 소셜 로그인

1. 로그인 화면에서 사용자가 소셜 로그인 버튼(네이버/카카오/구글/애플 중 하나)을 탭한다.
2. 모든 소셜 로그인 버튼을 비활성화하고 로딩 인디케이터를 표시한다.
3. 해당 소셜 SDK 인증 플로우를 호출한다.
4. 인증 성공 시 `socialSlct`, `socialKey`를 획득하고 로컬에 저장한다.
5. `POST /api/v1/auth/isRegistered`에 `{ socialSlct, socialKey }`를 전달한다.
6. 응답 code가 `"-1"`이면 `POST /api/v1/auth/login`을 호출하고, 응답의 `Set-Cookie`에서 JSESSIONID·remember-me를 암호화 저장소에 저장한 뒤 메인 화면으로 이동한다.
7. 응답 code가 `"1"`이면 `socialSlct`·`socialKey`를 온보딩 화면에 전달하며 이동한다.

## 경계 조건

- **서버 목록 중복 탭**: 3초 이내의 동일 서버 반복 탭은 두 번째 이후 요청을 차단한다.
- **로딩 중 추가 탭**: 소셜 로그인 처리 중 모든 소셜 로그인 버튼이 비활성화 상태이므로 중복 인증 요청이 발생하지 않는다.
- **쿠키 갱신**: `/auth/login` 이후 모든 API 응답에서 `Set-Cookie`가 오면 저장된 값을 덮어쓴다.
- **OAuth 사용자 취소**: OAuth 웹뷰·소셜 SDK 무대에서 사용자가 취소를 누르면 메시지 없이 조용히 버튼을 재활성화한다. 사용자의 의도적 선택이므로 에러 처리 대상이 아니다.
- **백그라운드 복귀 중 인증 유지**: 소셜 인증 진행 중 앱이 백그라운드로 전환 후 재전진되면 로딩 상태를 유지하고 SDK 결과 수신 시 정상 진행한다.
- **세션 만료 감지**: 메인 화면 진입 이후 API 호출에서 401 등 세션 만료 응답을 받으면 저장된 쿠키를 삭제하고 로그인 화면으로 이동한다. (이 동작은 본 Spec의 진입 플로우 외부에서 발생하지만 쿠키 삭제 규칙을 규정하기 위해 기록한다.)

# 6. 에러 처리 (Error Handling)

| 에러 유형 | 복구 전략 | 사용자 피드백 | 로깅 기준 |
|---|---|---|---|
| `/staff/list` 실패 | 무시 | 인라인 에러 | error |
| 소셜 SDK 인증 실패 | 버튼 재활성화 | 인라인 에러 | error |
| 소셜 SDK 인증 30초 타임아웃 | 버튼 재활성화 | 인라인 에러 | error |
| `/auth/isRegistered` 실패 | 버튼 재활성화 | 인라인 에러 | error |
| `/auth/isRegistered` 10초 타임아웃 | 버튼 재활성화 | 인라인 에러 | error |
| `/auth/login` 실패 | 버튼 재활성화 | 인라인 에러 | error |
| `/auth/login` 10초 타임아웃 | 버튼 재활성화 | 인라인 에러 | error |
| `/auth/login` code `"-11"` (계정 없음) | 버튼 재활성화 | 인라인 에러 | warn |
| 5xx 서버 에러 | 버튼 재활성화 | 인라인 에러 | error |
| 429 Rate Limit | 버튼 재활성화 | 인라인 에러 ("잠시 후 다시 시도해 주세요") | warn |
| 400 Bad Request | 버튼 재활성화 | 인라인 에러 (일반 메시지) | error |
| 네트워크 오류 (오프라인 등) | 버튼 재활성화 | 인라인 에러 | error |

> Wi-Fi↔셀룰러 전환 중 API 재시도 정책은 미정 — 보완 조건: 네트워크 복원력 설계 시.

# 7. 범위 (Scope)

## 포함

- 스플래시 화면의 서버 목록 조회·자동 선택·수동 선택
- API URL 설정과 로컬 저장
- 저장된 쿠키 기반 자동 로그인 판별
- 4개 소셜(네이버/카카오/구글/애플) SDK 인증 처리
- `/auth/isRegistered` → `/auth/login` 시퀀스와 쿠키 저장
- 가입 여부 분기(메인 또는 온보딩)
- 세션 만료 감지 시 쿠키 삭제와 로그인 화면 복귀 규칙

## 제외

- 이메일/비밀번호 로그인 (아이디+PW 방식)
- 온보딩 화면 자체의 동작 (별도 Spec)
- 계정 연동 해제 / 탈퇴
- 메인 화면 내부 동작

# 8. 제약 (Constraints)

- **접근성**: 모든 소셜 로그인 버튼은 공급자명을 포함한 스크린 리더 레이블을 제공한다.
- **보안**: 소셜 토큰·세션 쿠키는 iOS Keychain / Android EncryptedSharedPreferences에만 저장한다.
- **단일 세션**: 앱은 한 시점에 하나의 로그인 세션만 유지한다.

# 10. 기술 설계 (Technical Design)

## 10.1 구조

미정 — 개발자가 직접 기술 — 보완 조건: `feature` 파이프라인 3단계(계획) 진입 시. 스킬 규약상 섹션 10.1(모듈 분할·의존 그래프·아키텍처 패턴)은 프로젝트 고유 맥락 의존도가 가장 높아 AI 제안 대상이 아니며 개발자 판단이 필수다.

## 10.2 인터페이스

## 10.3 흐름

미정 — 상태 전이·알고리즘 — 보완 조건: `feature` 파이프라인 3단계(계획) 진입 시. 스킬 규약상 이 서브섹션도 AI가 `[AI 제안]` 태그와 함께 복수 안을 낼 수 있다. 호출 순서는 §5 행위 명세를 따른다.

# 13. 테스트 전략 (Test Strategy)

## 단위 테스트

- 쿠키 저장소: JSESSIONID/remember-me의 저장·조회·삭제 왕복이 정확히 동작한다. (AC-07 검증)
- 가입 여부 분기: `isRegistered` 응답 code에 따른 다음 액션 매핑이 정확하다. (AC-05, AC-06 검증)

## 통합 테스트

- 스플래시 → 자동 로그인 경로: 저장된 쿠키 존재 시 `/staff/list` 응답 수신 후 메인 화면 전이까지 동작한다. (AC-01 검증)
- 소셜 로그인 → 기존 회원 경로: 모의 SDK 성공 → `isRegistered` → `login` → 쿠키 저장까지의 API 시퀀스가 계약대로 호출된다. (AC-05, AC-07 검증)
- 상태 코드 처리: 5xx·429·400·OAuth 취소·백그라운드 복귀 각각의 기대 동작을 시뮬레이션한다. (AC-17~21 검증)

## E2E / UI 테스트

- 신규 사용자 첫 실행: 서버 선택 → 네이버 로그인 → 온보딩 이동. (AC-02, AC-06 검증)
- 재방문 사용자 재실행: 저장된 쿠키 상태에서 앱 실행 → 5초 이내 메인 도달. (AC-01, AC-15 검증)
- 세션 만료: 메인 화면 모의 401 트리거 시 로그인 화면 복귀. (AC-08 검증)
- 접근성: 스크린 리더 활성 상태에서 각 소셜 로그인 버튼 포커스 이동. (AC-16 검증)

- [ ] AC-02 [P1]: 저장된 API URL이 없는 상태에서 앱을 실행하면 서버 목록 UI가 표시된다.

- [ ] AC-04 [P1]: 소셜 로그인 버튼을 탭하면 모든 소셜 로그인 버튼이 비활성화되고 로딩 인디케이터가 표시된다.
- [ ] AC-05 [P1]: 소셜 인증 성공 후 `/auth/isRegistered` 응답 code가 `"-1"`이면 `/auth/login` 호출 성공 후 메인 화면으로 이동한다.
- [ ] AC-06 [P1]: 소셜 인증 성공 후 `/auth/isRegistered` 응답 code가 `"1"`이면 `socialSlct`와 `socialKey`가 전달된 상태로 온보딩 화면으로 이동한다.
