# Feature Specification: 스플래시 / 로그인 플로우

**Feature Branch**: `001-splash-login-flow`
**Created**: 2026-04-15
**Status**: Draft
**Input**: User description: "앱 실행 시 서버 선택 → 자동 로그인 판별 → 소셜 로그인 → 가입 여부 분기까지의 진입 플로우"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 재방문 사용자 자동 로그인 (Priority: P1)

이전에 로그인한 적이 있는 사용자가 앱을 재실행하면, 별도의 로그인 절차 없이 즉시 메인 화면으로 이동한다.

**Why this priority**: 매일 앱을 사용하는 재방문 사용자에게 가장 빈번히 발생하는 경로이며, 빠른 진입이 핵심 사용자 경험이다.

**Independent Test**: 이전 로그인 세션이 저장된 상태에서 앱을 실행하면 메인 화면으로 자동 이동하는지 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 이전에 로그인하여 세션이 저장된 상태이고 이전 서버 설정도 저장되어 있을 때, **When** 앱을 실행하면, **Then** 로그인 화면 없이 메인 화면으로 자동 이동한다.
2. **Given** 저장된 세션이 있고 앱이 메인 화면에 진입해 있을 때, **When** 서버가 세션 만료를 응답하면, **Then** 저장된 세션이 삭제되고 로그인 화면으로 이동한다.

---

### User Story 2 - 저장된 서버로 자동 연결 (Priority: P2)

이전에 사용했던 서버 정보가 저장된 경우, 서버 선택 화면 없이 자동으로 해당 서버에 연결한다.

**Why this priority**: 재방문 사용자의 자동 로그인(P1)을 가능하게 하는 필수 선행 조건이다.

**Independent Test**: 저장된 서버 설정이 있는 상태에서 앱 실행 시 서버 목록 UI가 나타나지 않고 자동으로 연결되는지 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 이전에 사용했던 서버 정보가 저장된 상태에서, **When** 앱을 실행하면, **Then** 서버 선택 화면 없이 저장된 서버로 자동 연결된다.
2. **Given** 서버 목록 조회에 실패했을 때, **When** 앱이 시작 화면에 있으면, **Then** 오류 메시지가 표시된다.

---

### User Story 3 - 서버 선택 후 소셜 로그인 (Priority: P2)

저장된 서버나 세션이 없는 사용자가 서버 목록에서 서버를 선택하고 소셜 계정으로 로그인한다.

**Why this priority**: 신규 사용자와 서버를 변경하는 기존 사용자에게 필요한 핵심 로그인 경로이다.

**Independent Test**: 저장된 서버/세션 없이 앱을 실행하여 서버 선택 → 소셜 로그인 → 메인 화면 진입까지 전체 흐름을 독립적으로 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 저장된 서버 설정이 없을 때, **When** 앱을 실행하면, **Then** 연결 가능한 서버 목록이 표시된다.
2. **Given** 서버 목록이 표시된 상태에서, **When** 사용자가 서버를 선택하면, **Then** 저장된 로그인 세션이 초기화되고 로그인 화면으로 이동한다.
3. **Given** 로그인 화면에서, **When** 소셜 로그인 버튼(네이버/카카오/구글/애플)을 탭하면, **Then** 해당 소셜 인증이 시작되고 모든 로그인 버튼이 비활성화된다.
4. **Given** 소셜 인증이 성공하고 기존 회원인 경우, **When** 인증이 완료되면, **Then** 세션이 저장되고 메인 화면으로 이동한다.
5. **Given** 소셜 인증 중 오류가 발생했을 때, **When** 인증이 실패하면, **Then** 오류 메시지가 표시되고 로그인 버튼이 다시 활성화된다.

---

### User Story 4 - 미등록 사용자 온보딩 연결 (Priority: P3)

소셜 로그인에 성공했지만 서비스에 등록되지 않은 신규 사용자를 온보딩 화면으로 안내한다.

**Why this priority**: 신규 사용자 유입 경로로 중요하지만, 기존 사용자 로그인 경로(P2)가 먼저 완성되어야 한다.

**Independent Test**: 서비스에 등록되지 않은 소셜 계정으로 로그인 시도하여 온보딩 화면으로 이동하는지 독립적으로 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 소셜 인증에 성공했지만 서비스 미등록 사용자인 경우, **When** 인증이 완료되면, **Then** 온보딩 화면으로 이동하며 소셜 인증 정보가 함께 전달된다.

---

### Edge Cases

- 서버 목록 조회가 실패하면 어떻게 되는가? → 오류 메시지 표시 (앱 재실행으로 재시도)
- 소셜 인증 도중 네트워크가 끊기면 어떻게 되는가? → 오류 메시지 표시, 로그인 버튼 재활성화
- 로그인 중 동일 소셜 버튼을 연속으로 탭하면 어떻게 되는가? → 첫 번째 요청 처리 중 모든 버튼 비활성화로 중복 요청 방지
- 저장된 서버에서 수동으로 다른 서버를 선택하면 어떻게 되는가? → 기존 세션 삭제 후 로그인 화면으로 이동
- 메인 화면 진입 후 서버에서 인증 실패를 응답하면 어떻게 되는가? → 저장된 세션 삭제 후 로그인 화면으로 이동
- 가입 확인 요청은 성공했으나 이후 로그인 요청이 실패하면 어떻게 되는가? → 오류 메시지 표시, 로그인 버튼 재활성화

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 앱은 MUST 서버 시작 시 연결 가능한 서버 목록을 조회해야 한다
- **FR-002**: 앱은 MUST 이전에 선택한 서버 정보가 있을 경우 자동으로 해당 서버에 연결해야 한다
- **FR-003**: 앱은 MUST 저장된 서버 정보가 없을 경우 서버 목록을 화면에 표시해야 한다
- **FR-004**: 앱은 MUST 사용자가 서버 목록에서 서버를 선택할 수 있어야 한다
- **FR-005**: 시스템은 MUST 사용자가 수동으로 서버를 선택하면 저장된 로그인 세션을 삭제해야 한다
- **FR-006**: 시스템은 MUST 저장된 로그인 세션이 존재할 경우 로그인 화면 없이 메인 화면으로 이동해야 한다
- **FR-007**: 앱은 MUST 네이버, 카카오, 구글, 애플 소셜 로그인을 지원해야 한다
- **FR-008**: 시스템은 MUST 소셜 인증 처리 중 중복 요청을 방지해야 한다
- **FR-009**: 시스템은 MUST 소셜 인증 성공 후 해당 사용자의 서비스 등록 여부를 확인해야 한다
- **FR-010**: 시스템은 MUST 기존 회원의 로그인 성공 시 사용자 세션을 로컬에 영속 저장해야 한다
- **FR-011**: 시스템은 MUST 기존 회원의 로그인 성공 후 메인 화면으로 이동해야 한다
- **FR-012**: 시스템은 MUST 미등록 사용자를 소셜 인증 정보와 함께 온보딩 화면으로 안내해야 한다
- **FR-013**: 시스템은 MUST 서버로부터 인증 실패 응답 수신 시 저장된 세션을 삭제하고 로그인 화면으로 이동해야 한다
- **FR-014**: 앱은 MUST 모든 오류 상황에서 사용자에게 명확한 오류 메시지를 표시해야 한다

### Key Entities

- **서버(Server)**: 연결 가능한 서비스 환경. 사용자에게 표시되는 이름과 접속 주소를 가진다.
- **사용자 세션(User Session)**: 로그인 성공 후 로컬에 저장되는 인증 상태. 앱 재시작 시에도 유지되며, 서버 변경 또는 인증 만료 시 삭제된다.
- **소셜 계정(Social Identity)**: 소셜 인증 제공자(네이버/카카오/구글/애플)와 해당 제공자의 사용자 식별자로 구성된다.
- **회원(Member)**: 서비스에 등록된 사용자. 소셜 계정으로 인증하여 접근한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 저장된 세션이 있는 사용자는 앱 실행 후 3초 이내에 메인 화면에 도달한다
- **SC-002**: 소셜 로그인 전체 흐름(버튼 탭 → 메인 화면 진입)을 30초 이내에 완료할 수 있다
- **SC-003**: 기존 회원의 100%가 로그인 성공 후 메인 화면으로 정확하게 이동한다
- **SC-004**: 미등록 사용자의 100%가 소셜 인증 성공 후 온보딩 화면으로 정확하게 이동한다
- **SC-005**: 로그인 오류 발생 시 앱 재시작 없이 동일 화면에서 재시도가 가능하다
- **SC-006**: 연속 탭 시 중복 로그인 요청이 발생하지 않는다

## Assumptions

- 사용자는 네이버, 카카오, 구글, 애플 중 하나 이상의 소셜 계정을 보유하고 있다
- 서버 목록은 서비스 운영팀이 관리하며 사용자가 임의로 추가할 수 없다
- 세션 유효성은 앱에서 직접 판별하지 않으며, 메인 화면에서 서버의 인증 실패 응답으로만 감지한다
- 자동 로그인은 세션 유효성 검증 없이 저장된 세션 존재 여부만으로 판단한다
- 여러 서버 환경(개발, 스테이징, 운영 등)이 존재하며 앱은 이를 선택할 수 있어야 한다
- 온보딩 화면의 구체적인 내용은 본 Spec의 범위 밖이다 (별도 Spec에서 정의)
- 서버 이미지 서버 URL 등 부가 정보는 서버 선택 시 함께 저장된다
